# 实现方案文档模板

Phase 3 写入 `docs/plans/YYYY-MM-DD-<feature>-实现方案.md` 时使用以下结构（可按功能删减小节，但标题尽量保留便于对照）。

```markdown
# <FeatureId / 名称> 实现方案

> 日期：YYYY-MM-DD  
> 状态：待实现 / 实现中 / 已完成  
> 关联：PRD 章节 / 相关笔记链接  

## 1. 已确认的决策

### 1.1 UX 方案（用户选定）
- 选定：方案 X — …
- 用户主路径：…
- 未选方案（归档）：方案 Y — 一句话原因

### 1.2 技术方案（用户选定）
- 选定：路线 Z — …
- 关键技术点：…
- 未选路线（归档）：…

## 2. 范围

### 2.1 In Scope
- …

### 2.2 Out of Scope
- …

## 3. 用户操作对照

| 步骤 | 用户动作 | 界面反馈 | 失败/边界 |
|------|----------|----------|-----------|
| 1 | … | … | … |

## 4. 技术设计

### 4.1 架构与数据流
（简图或步骤列表）

### 4.2 API
| 方法 | 路径 | 请求 | 响应 | 鉴权 |
|------|------|------|------|------|
| POST | … | … | … | … |

### 4.3 数据 / 存储
- 新表或字段：…（无则写「无」）
- 复用已有：…

### 4.4 前端
- 入口组件：…
- 新增/修改文件：…
- 状态与文案（i18n key 若已知）：…

### 4.5 后端
- Module / Service / Controller：…
- 复用：…
- 新建：…

### 4.6 错误与边界
- …

## 5. 实现任务清单（对照编码用）

- [ ] Task 1：…（涉及文件：…）
- [ ] Task 2：…
- [ ] Task 3：测试 / 手测：…

## 6. 验收标准

- [ ] …
- [ ] …

## 7. 测试计划

- 单测：…
- e2e / 手测：…

## 8. 风险与后续

- 风险：…
- 明确延后：…
```

## 写作要求

- 文件路径写到**具体文件**，不要写「相关 service」。
- 任务粒度：每个 Task 应能在一轮实现中完成并核对验收项。
- 与 PRD 冲突时：以用户在 Phase 1/2 的选择为准，并在文档「已确认的决策」里写明。
