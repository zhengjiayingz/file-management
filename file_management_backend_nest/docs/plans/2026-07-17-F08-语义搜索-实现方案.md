## 1. 已确认的决策

### 1.1 UX 方案（用户选定）

- **选定：方案 B + 策略 C**
  - **B：** 独立「智能搜索」入口（非改 FileFilterBar 文件名筛选），打开专用搜索弹层/抽屉。
  - **C（半自动）：** 语义结果 **仅覆盖** `DocumentIndexJob.status = ready` 的文件；**本批不做**上传自动建索引；空态说明「未建索引则语义搜不到」，并引导去预览里「建立索引」。
- **用户主路径：**
  1. 在网盘顶栏（`GlobalHeader`）点「智能搜索」
  2. 输入自然语言（如「Nest 迁移相关笔记」）→ 搜索
  3. 看到按相关度排序的 **文件列表**（可含短摘录）
  4. 点击结果 → 打开预览 / 定位到所在目录（与现有文件打开行为对齐）
  5. 无结果或库内几乎无索引 → 空态文案 + 引导建索引
- **未选方案（归档）：**
  - UX A（筛选栏加模式切换）— 用户更倾向独立入口。
  - UX C（输入框自动语义优先）— 行为难解释。
  - 策略 A/D（上传全自动 / 选择性自动索引）— 成本高，延后到 F-18 配额后再议。

### 1.2 技术方案（用户选定）

- **选定：路线 Y1** — 独立 `GET /api/files/search` + 前端智能搜索弹层；与 `GET /api/files?q=` 文件名搜索解耦。
- **关键技术点：** `embedOne`、按文件聚合 chunk 相似度、`FilesQueryController` 静态路由顺序、`GlobalHeader` 入口 + Dialog。
- **未选路线（归档）：**
  - Y2 扩 `GET /files?mode=semantic` — 与 parentId/列表筛选缠在一起。
  - Y3 绕 RAG/KB chat 拼文件列表 — 链路不当。

## 2. 范围

### 2.1 In Scope

- `GET /api/files/search?q=`：对当前用户已索引文件做语义检索，返回文件列表（含 `score`、可选 `excerpt`）。
- 响应附带轻量元信息：如 `indexedFileCount`（便于空态区分「没索引」vs「有索引但无匹配」）。
- 前端：`GlobalHeader`（或文件页 header 右侧）入口 + `SemanticSearchDialog`（名称可微调）。
- i18n：zh-CN / zh-TW / en-US。
- 单测：聚合打分 / 空库 / 非法 q。

### 2.2 Out of Scope

- 上传自动建索引、批量一键建索引。
- F-15 Hybrid（向量 + FULLTEXT 融合）。
- 回收站语义搜索。
- 改 FileFilterBar 的文件名 `q` 行为（保留原样）。
- 向量库 / Redis 缓存（MVP 内存扫描即可；有上限保护）。

## 4. 技术设计

### 4.1 架构与数据流

```
用户输入 q
  → GET /api/files/search?q=
  → 鉴权 + 校验 q（trim，长度上限）
  → 查 DocumentIndexJob：userId + status=ready → fileIds + indexedFileCount
  → 若 fileIds 空：返回 { items: [], indexedFileCount: 0 }
  → 拉这些文件的 DocumentChunk（id, userFileId, content, embedding）
  → embedOne(q)
  → 对每个 chunk 算 cosine；按 userFileId 聚合（推荐 max(score)）
  → 取 Top-N 文件（如 N=20），附带该文件最高分 chunk 的 excerpt
  → 拼 FileItem 必要字段（id, fileName, fileType, …）返回
```

### 4.2 API

| 方法 | 路径 | 请求 | 响应 | 鉴权 |
|------|------|------|------|------|
| GET | `/api/files/search` | Query：`q`（必填）；可选 `limit`（默认 20，上限 50） | 见下 | JWT |

**成功响应（建议）：**

```json
{
  "success": true,
  "data": {
    "items": [
      {
        "id": 123,
        "fileName": "nest-migration.md",
        "fileType": "md",
        "mimeType": "text/markdown",
        "fileSize": 1024,
        "parentId": null,
        "score": 0.82,
        "excerpt": "……与 Nest 迁移相关的笔记……",
        "chunkIndex": 3
      }
    ],
    "indexedFileCount": 12,
    "q": "Nest 迁移"
  }
}
```

说明：

- **不传** `mode=semantic`：本端点专用于语义；文件名搜索继续用 `GET /api/files?q=`。
- 路由必须声明在 `FilesQueryController` 的 `@Get(':id')` **之前**（与 `batch/download-zip` 同类静态路径）。

### 4.3 数据 / 存储

- 新表或字段：**无**
- 复用：`DocumentIndexJob`、`DocumentChunk.embedding`、`UserFile`

### 4.4 前端

| 项 | 路径 / 说明 |
|----|-------------|
| 入口 | `GlobalHeader` 右侧增加按钮（文件主页等已挂 Header 的视图可见即可；至少 `views/index/index.vue`） |
| 弹层 | 新建如 `components/SemanticSearchDialog/index.vue`：输入、搜索、结果列表、空态 |
| API | `api/file.ts` 增加 `searchFilesSemantic({ q, limit? })` → `GET /files/search` |
| 打开文件 | 复用 index 页已有预览/打开逻辑（emit 或注入回调 / 提供 `fileId` 给现有 handler） |
| i18n | `semanticSearch.*`（title / placeholder / emptyNoIndex / emptyNoMatch / open 等） |

**不改：** `FileFilterBar`、`FileFilterState.q` 的文件名包含搜索。

### 4.5 后端

| 项 | 路径 / 说明 |
|----|-------------|
| Service | 新建 `src/files/query/files-search.service.ts`（或 `files-semantic-search.service.ts`） |
| Controller | `src/files/query/files-query.controller.ts` 增加 `@Get('search')`，注入 SearchService |
| Module | `src/files/files.module.ts` 注册 provider |
| 复用 | `embedOne`（`embedding.provider.ts`）、`cosineSimilarity`（`similarity.util.ts`） |
| 参考 | `KnowledgeBasesChatService.retrieveTopChunks` 的多文件 chunk 扫描，但输出改为 **按文件聚合列表**，不调 LLM |

**聚合规则（MVP）：** 每个 `userFileId` 取 **max(chunk cosine)** 作为 `score`；`excerpt` 取该最高分 chunk 正文截断（如 160 字）。

**性能保护（MVP）：**

- 仅 `ready` 文件；
- `limit` 默认 20；
- 若 ready 文件数或 chunk 总数过大：可先限制参与检索的文件数（如最近索引的 200 个）并在实现方案验收中写明；超限时仍返回结果 + 不阻塞（可选日志）。

### 4.6 错误与边界

| 情况 | 处理 |
|------|------|
| `q` 空 / 过长 | 400 |
| 无 ready 索引 | 200 + `items=[]` + `indexedFileCount=0`（非 4xx，方便前端空态） |
| Embedding 服务失败 | 502/500 + 友好 message |
| 已删文件 | 聚合时 join `UserFile isDeleted=false`，过滤掉 |
