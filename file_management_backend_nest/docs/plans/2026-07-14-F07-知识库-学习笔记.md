## 问答一：F-07 具体是要做一个什么东西？

### 问题

> 我们先做 F07，先给我捋一遍 F07 具体是要做一个什么东西。

### 一句话

**F-07 = 多文档知识库问答**，不是 F-06 那种「单篇论文知识卡片」。

用户建一个「知识库」，从网盘往里塞多个已索引文件，然后**对着整个库提问**；回答要能**跨文件**，并带上 **引用（哪个文件、哪一段）**，还要支持**多轮会话**。

产品叙事：从「打开一个文件问 AI」升级成「个人知识管理 / 小型 Dify」。

### 和现有能力的关系

| 已有 | 作用 |
|------|------|
| **F-01** 索引 | 文件切块 + embedding，存在 `DocumentChunk` |
| **F-02** 单文件 RAG | 对单个 `fileId` 提问 |
| **F-06** 学术知识卡片 | 对 `paper` 抽结构化字段，预览侧栏展示 |

**F-07 不重新做 embedding**：知识库只维护「这个库里有哪些 fileId」。检索时把范围从「一个文件」扩成「库里所有文件」，复用同一套 chunk。

面试故事就是：**文件只 index 一次，知识库是多文件检索视图**。

### 产品长什么样

1. **知识库列表**：创建 / 改名 / 删除  
2. **库详情**：左边文件列表（增删），右边对话  
3. **聊天**：流式回答 + citations；点 citation 能跳到对应文件预览  
4. **约束**：库里文件要先索引好；未全 ready 时提示先索引，不能硬问

### 数据与 API（PRD 定的）

**表**

- `KnowledgeBase` — 库本身  
- `KnowledgeBaseItem` — 库 ↔ 文件  
- `KnowledgeBaseSession` / `KnowledgeBaseMessage` — 多轮会话与历史  

**接口**

- `POST/GET/PATCH/DELETE /api/knowledge-bases`  
- `GET/POST/DELETE /api/knowledge-bases/:id/items`  
- `GET /api/knowledge-bases/:id/index-status`  
- `GET /api/knowledge-bases/:id/sessions`  
- `GET /api/knowledge-bases/:id/sessions/:sessionId/messages`  
- `DELETE /api/knowledge-bases/:id/sessions/:sessionId`  
- `POST /api/knowledge-bases/:id/chat`（流式 `text/plain` + 响应头 `X-Session-Id` / `X-Citations`）  

**引用结构**大致是：

```ts
{ fileId, fileName, chunkIndex, excerpt }
```

检索逻辑：`WHERE userFileId IN (kb 里的文件)` → Top-K → 再喂给 LLM。

### 验收（做完要能证明这些）

1. 两个已索引文件进同一库，问跨文件问题 → 有答案 + **至少 1 个 source**  
2. 会话历史能回看  
3. 未索引文件进库时，提示先索引，不能假装能答  

工期 PRD 估 **7～10 天**（S16），难度 4、价值 5。

### 明确不做（本项范围内）

- **F-08** 语义搜文件（返回文件列表，不是对话）  
- **F-13** 多论文对比表（依赖 F-07，后面再做）  
- **F-16** 插件剪藏进库  
- **F-06** 那套 academic 卡片 / `GET .../ai/knowledge`  

### 落地时大致 6 步（PRD「怎么做」）

1. KB CRUD + items 增删  
2. 聚合子文件 index 状态（全 ready 才可问）  
3. 把现有 rag-ask 泛化成「多 fileId 范围检索」  
4. citations 定好结构  
5. 前端：列表 + 详情（左文件 / 右对话）  
6. citation 点击跳转预览

## 问答二：大致实现步骤与技术点清单

### 问题

> 先给我拆一下，F07 的大致实现步骤，以及需要用到哪些技术点？

### 大致实现步骤

#### 阶段 0：定边界（半天）

- MVP 是否含会话持久化（PRD 有 Session/Message；也能先做客户端带 history、后端暂不落库）
- citations 怎么回传：流结束后 JSON、还是 SSE 双通道（现有 `pipeTextStreamToResponse` 是纯文本流，这是 F-07 和 F-02 的主要差异）
- 库内文件上限（例如 20）、Top-K 跨库还是按文件配额

#### 阶段 1：数据模型（1 天）

1. Prisma 加 4 张表：`KnowledgeBase` / `KnowledgeBaseItem` / `KnowledgeBaseSession` / `KnowledgeBaseMessage`
2. Item 只存 `knowledgeBaseId + userFileId`，**不**复制 `DocumentChunk`
3. 权限：所有查询带 `userId`，只能操作自己的库和文件
4. migrate + Nest module（建议独立 `knowledge-bases/`，别塞进 `files/ai` 单文件里膨胀）

#### 阶段 2：KB CRUD + 文件进出（1～2 天）

1. `POST/GET/PATCH/DELETE /api/knowledge-bases`
2. `POST .../items`（加）+ `DELETE .../items/:fileId`（删）
3. 校验：文件存在、未删除、属于当前用户；是否已在库中
4. **聚合索引状态**：查 items → join `DocumentIndexJob`  
   - `readyCount / total`  
   - 「可否提问」：`all ready`（PRD 验收）  
   - 未索引给出明确 409 + 未 ready 的 fileName 列表

#### 阶段 3：多文件检索泛化（核心，2 天）

从现有 `FilesAiRagService.ragAsk` 抽出可复用检索：

| 现有 F-02 | F-07 要改成 |
|-----------|-------------|
| `where: { userFileId: fileId }` | `userFileId IN (kb.items)` |
| prompt 标一个文件名 | 片段带 `【文件名】` / `fileId` |
| 只返文本流 | 还要带走 citations |

具体流：

1. 校验 KB 归属 + items 非空 + 全员 `index=ready`
2. `embedOne(question)`（复用 `embedding.provider`）
3. 拉多文件 chunks（注意内存：文件多时不要无脑 `findMany` 全库，可限 Top 文件数或分批）
4. `topKByEmbedding`（复用 `similarity.util`）
5. 组装 selected chunks → system prompt
6. `streamText` + 流式写回

**citations**：检索命中时就能生成，不必等模型编造：

```ts
{ fileId, fileName, chunkIndex, excerpt } // excerpt 可截断 content 前 N 字
```

回传方式建议 MVP：

- 方案 A：`X-Citations` / 响应头放 base64 JSON（简单但脆弱）
- 方案 B：先非流式 JSON `{ answer, citations }`（最快验收，再改流式）
- 方案 C：自定义 SSE：`event: citations` + `event: text`（更完整，工期更长）

#### 阶段 4：会话（1～2 天）

1. `POST .../sessions` 或 chat 时懒创建 session
2. 每轮落库 user/assistant message
3. `GET .../sessions`、`GET .../sessions/:id/messages`
4. chat 入参：`sessionId?` + `question`；历史可从 DB 读，也可继续像 F-02 一样前端传 `messages`（二选一，别两套并行太久）

#### 阶段 5：前端（2～3 天）

1. 路由/入口：「知识库」（列表页）
2. 详情：左文件列表（从网盘选择器加文件）+ 右对话（可大量复用 `DocumentAiPanel` 聊天 UI）
3. 顶部状态条：索引未完成时禁用发送 + 文案
4. 消息下展示 citations → 点击跳转文件预览（现有预览路由 + `fileId`）
5. API 层仿 `streamRagAsk`，再处理 citations 协议

#### 阶段 6：测试与验收（1～2 天）

1. 单测：检索 scope（IN 多个 fileId）、全未 ready → 409、空库 → 4xx
2. e2e：建库 → 加 2 个 seed ready 文件 → chat → 有 citations
3. 手工：真正跨文档问题能点到对的文件

### 会用到的技术点

#### 已有、直接复用

| 技术点 | 你们现在的位置 |
|--------|----------------|
| Prisma / MySQL | `schema.prisma`、`DocumentChunk.embedding`（Json） |
| Nest Module / Controller / Guard | 与 `files/ai` 同模式 |
| Embedding | `embedOne` |
| 向量相似度 Top-K | `topKByEmbedding`（内存余弦，无向量库） |
| LLM 流式 | Vercel AI SDK `streamText` + `pipeTextStreamToResponse` |
| 单文件 RAG 校验 | index ready、abort on client close |
| 前端 fetch 读流 | `src/api/ai.ts` → `readTextStream` |

#### F-07 新增/加深

| 技术点 | 说明 |
|--------|------|
| **多实体资源建模** | KB 1-N Item；Session 1-N Message |
| **权限边界** | 库、文件、会话都属同一 `userId` |
| **状态聚合** | 多文件 `DocumentIndexJob.status` 汇总 |
| **多文件检索 scope** | `IN (...)` + prompt 标明来源文件 |
| **Citation / 溯源** | 检索结果结构化回传，点选跳转 |
| **流式 + 元数据双通道** | 比 F-02 多一步协议设计 |
| **会话持久化** | 多轮历史落库（若做完整 PRD） |
| **前端双栏布局** | 文件管理 + 对话，接网盘选文件 UX |
| **规模意识（MVP 可浅）** | chunk 一次加载的内存上限；后期才迁 pgvector |

#### 刻意先不做的技术

- 独立向量数据库（Qdrant/pgvector）— PRD MVP 继续 MySQL Json + 内存 Top-K  
- Agent / 工具调用 / 插件剪藏（F-16）  
- 论文对比表（F-13）  
- F-06 学术卡片逻辑（名字像 knowledge，产品完全不同）

### 推荐工期切片（对应 7～10 天）

```
D1     Schema + KB CRUD
D2     Items + 索引状态聚合
D3–D4  多文件 RAG chat（建议先非流式或「流文本 + 固定 citations」跑通）
D5     Session/Message
D6–D8  前端列表/详情/对话/citation 跳转
D9–D10 测试 + 修体验
```

### 和 F-02 的本质差别（记住这一句就够）

**F-02**：一个 `fileId` → 搜它的 chunks → 流式答。  
**F-07**：一个 `kbId` → 搜它下面一堆 `fileId` 的 chunks → 流式答 + **带来源** +（可选）**会话历史落库**。

核心新活是「资源管理 + 检索范围 + citation 协议 + 前端双栏」，LLM/embedding 本身几乎不用新技术。

### 动手前建议先定的两件产品决策

1. citations 用「纯流 + 头/尾 JSON」还是「自定义 SSE」  
2. 会话是否 MVP 必做，还是 v1 只做可追问、历史不持久化

## 问答三：新增技术点详细解释（本文重点）

### 问题

> 新增的技术点都是什么，详细给我解释一下，我搞不懂这些东西。

### 说明

下面只讲 **F-07 相对 F-02「新」的那些概念**。已经会的（Embedding、Top-K、`streamText`）不再展开。

---

### 1. 多实体资源建模

**是什么：** 不只一张表，而是好几类「东西」互相挂钩。

F-02 很简单：用户 → 一个文件 → 对这个文件提问。  
F-07 要表达的是一整条故事：

```
知识库（KnowledgeBase）
  ├── 里面有哪些文件（Item）：库 A 包含 文件1、文件2
  └── 聊过几次（Session）
        └── 每一句聊了什么（Message）
```

- **知识库**：一个文件夹/专题，比如「考研英语笔记」  
- **Item**：库和文件的关系（中间表），只记「这个库挂了哪些 fileId」  
- **Session**：一次完整对话（今天下午那次）  
- **Message**：这一次里的每一句（用户问 / AI 答）

**为什么叫「建模」：** 用数据库表把真实业务关系固定下来，后面 API、前端都围绕这些表转。  
**和 F-06 的 knowledge 无关：** F-06 是「一篇论文的结构化卡片」；这里的知识库是「一堆文件的容器」。

---

### 2. 权限边界

**是什么：** 保证「别人的库 / 别人的文件，你碰不到」。

每个接口都要想清楚：

- 这个知识库是不是当前登录用户的？  
- 往库里加的文件，是不是也是这个用户的？  
- 去读某个 session，是不是属于这个用户的库？

不能只校验「文件存在」，还要校验 **userId 一路对得上**。否则容易出现：知道别人的 `kbId` 就能问别人的资料。

F-02 也有「只能问自己的 fileId」；F-07 多了一层「库」，所以链路更长，更容易漏。

---

### 3. 状态聚合

**是什么：** 库里有很多文件，每个文件各自有索引状态；问库之前，要先把这些状态合起来看。

每个文件都有自己的索引：`pending` / `indexing` / `ready` / `failed`（`DocumentIndexJob`）。

F-02：只看 **这一个** 文件是不是 `ready`。  
F-07：库里可能 5 个文件，可能出现：

- 3 个 ready、1 个还在 indexing、1 个 failed  

**聚合**就是汇总成一眼能用的结论，例如：

- `readyCount = 3, total = 5`  
- `canChat = false`（PRD：要全部 ready 才能问）  
- 并告诉前端：「哪些文件还没索引好」

前端才能显示：「还有 2 个文件未就绪，暂不可提问」，而不是一问就炸或答得莫名其妙。

---

### 4. 多文件检索 scope（检索范围）

**是什么：** AI 答题前，先从向量库/chunk 里「搜相关段落」；scope = **允许搜哪些范围**。

F-02：

```
只在「当前这个 fileId」的 chunks 里搜 Top-K
```

F-07：

```
在「这个知识库挂上的所有 fileId」的 chunks 里搜 Top-K
```

SQL/Prisma 上就是大致从：

`where userFileId = 123`  

变成：

`where userFileId in (123, 456, 789)`  

**为什么还要在 prompt 里写文件名：**  
搜出来的片段可能来自不同文件。模型如果不知道这段来自哪篇，容易把两篇内容揉在一起胡说。所以系统提示里要标明：

> 片段来自《论文A》…  
> 片段来自《实验报告B》…

这样跨文档问答才有意义。

---

### 5. Citation / 溯源

**是什么：** 回答旁边列出「我依据了哪些来源」，并可点进去核对原文。

普通聊天机器人：只给一段话。  
带 citation 的：还会说「依据文件 X 第 N 段」，并附上摘录。

F-07 期望结构类似：

```ts
{
  fileId: 12,
  fileName: "Nest迁移笔记.md",
  chunkIndex: 3,
  excerpt: "……当时用 BullMQ 做索引……"
}
```

注意：citation **主要来自检索命中的 chunk**，不是让模型自己编「引用了第几页」。  
流程是：

1. 向量检索选出片段  
2. 用这些片段生成回答  
3. 同时把这些片段整理成 citations 返回前端  
4. 用户点击 → 打开对应 `fileId` 的预览

这是「可信感」和面试里常说的 **RAG with source grounding**。

---

### 6. 流式 + 元数据双通道

**是什么：** 回答一个字一个字往外蹦（流式），但 citations 是结构化数据；两种信息怎么一起传给前端。

你们现在的 F-02：

- 后端：`pipeTextStreamToResponse` → 纯文本流  
- 前端：一边读一边把文字拼到气泡里  

问题：citations 是 `{ fileId, ... }` 这种 JSON，**塞不进「纯文字流」**里（硬塞进去前端分不清哪是答案哪是元数据）。

所以要设计「两条通道」或「两段协议」：

| 方案直觉 | 做法 |
|----------|------|
| A | 文字照常流；citations 放响应头 / 流结束后另包一段 |
| B | 先不流式，一次返回 `{ answer, citations }`（最好懂、最快验收） |
| C | SSE 事件：`event: text` 一块块字；`event: citations` 一次发来源 |

**「双通道」** = 用户看到的答案文字，和机器用的结构化来源，**分开发、分别解析**。  
这是 F-07 比 F-02 多出来的协议设计，不是新的 AI 模型技术。

---

### 7. 会话持久化

**是什么：** 对话历史写进数据库，刷新页面、过两天还能接着看。

F-02 现状大概是：

- 多轮追问：前端把最近几轮 `messages` 放进请求体  
- 关页面 → 历史没了（或只在前端内存里）

**持久化**就是：

- 选/建一个 `Session`  
- 用户问一句 → 存一条 Message  
- AI 答一句 → 再存一条  
- 下次 `GET` 把历史拉出来展示  

好处：可回溯、「会话历史可查看」（PRD 验收项）。  
代价：多两张表、多几个接口、要注意「历史太长截断再送模型」。

MVP 也可以先不做落库，只做当次可追问；但完整 F-07 一般要求持久化。

---

### 8. 前端双栏布局

**是什么：** 界面左右分工，不是再塞进「单文件预览旁的小聊天框」。

典型：

```
┌──────────────┬─────────────────────┐
│ 知识库文件列表 │  对话区              │
│ + 添加/移除   │  输入框 / 气泡       │
│ 索引状态提示  │  citation 列表       │
└──────────────┴─────────────────────┘
```

左边是 **资源管理**（哪些文件在库里），右边是 **对话**。  
还要接「从网盘选文件加入知识库」——这是文件管理 UX，不是纯 AI 组件。

单文件预览里的 `DocumentAiPanel` 可以复用聊天样式，但页面职责变了：从「问当前这篇」变成「管一个库并问整个库」。

---

### 9. 规模意识（MVP 可浅做）

**是什么：** 一开始就要知道「库一大」时哪里会疼，即使现在先用笨办法。

F-02：一个文件几十～几百 chunk，全部载入内存做相似度，通常够用。  
F-07：10 个文件 × 每文件 200 chunk = 2000 条 embedding 进内存比一次，可能还能扛；再大就会慢、吃内存。

所以要有「规模意识」：

- MVP：限制「一个库最多 N 个文件」  
- 以后：换向量数据库（pgvector 等），按向量索引检索，而不是每次全表扫

**现在不必上向量库**；只要知道当前方案的天花板，避免设计成「无限加文件还不设限」。

---

### 串起来看一遍（请求发生时）

```
你问知识库：「这两篇对 Nest 迁移的说法差在哪？」
        │
        ▼
① 权限：这是你的库吗？
② 状态聚合：库里文件都 ready 了吗？没 → 提示先索引
③ 多文件 scope：在库内所有文件的 chunks 里向量检索
④ 生成 citations：命中了哪些文件的哪几段
⑤ 把片段 + 历史喂给 LLM，流式吐出回答（文字通道）
⑥ 同时把 citations 交给前端（元数据通道）
⑦ 会话持久化：把问句和答句写入 Session
⑧ 前端双栏：左边文件、右边气泡 + 可点击的来源
```

---

### 和「旧技术」的关系（帮你对齐）

| 你已经会的 | F-07 怎么用它 |
|------------|----------------|
| Embedding | 问题仍要 embed 一次 |
| Top-K 相似度 | 仍用，只是候选 chunk 来自多个文件 |
| streamText | 仍用，多解决「citations 怎么附带」 |
| DocumentChunk | **不新建一份**，知识库只引用 fileId |

所以 F-07 难的不是「换一个更强的模型」，而是 **产品建模 + 权限 + 状态 + 检索范围 + 引用展示 + 会话/前端**。
