## 问答一：F-06 会学到哪些知识点？

### 问题

F-06 学到的知识点给我详细解释一下是什么东西。

### 背景：已掌握 vs 将要学

| 已掌握 | 来自 |
|--------|------|
| 流式 LLM、`AbortController` | F-00 |
| 离线索引、BullMQ、分块 | F-01 |
| Embedding、Cosine、Top-K RAG | F-02 |
| Map-Reduce 摘要、`generateObject` + Zod | F-03 |
| PDF 文本提取、多格式 pipeline | F-05 |

### F-06 在做什么？

已有能力：

| 功能 | 回答的问题 |
|------|-----------|
| F-02 RAG | 「论文里某句话在哪？某细节是什么？」 |
| F-03 摘要 | 「这篇论文/小说整体讲什么？」 |

F-06 回答第三个问题：

> **「这篇论文有哪些结构化知识点？贡献是什么？方法是什么？关键发现有哪些？局限在哪？」**

输出不是自由文本，而是**固定字段的知识卡片 JSON**，前端用卡片 UI 展示，存入 `DocumentKnowledge` 表，查看时读库而非实时跑 LLM。

示例结构（设计文档 schema）：

```json
{
  "title": "Attention Is All You Need",
  "researchQuestion": "能否只用注意力机制做序列建模？",
  "contributions": ["提出 Transformer", "..."],
  "methodology": {
    "approach": "Self-Attention",
    "dataset": "WMT 英德翻译",
    "metrics": ["BLEU"]
  },
  "keyFindings": [
    {
      "claim": "Transformer 在翻译任务上优于 RNN",
      "evidence": "Table 2",
      "section": "results"
    }
  ],
  "definitions": [
    { "term": "Multi-Head Attention", "definition": "...", "section": "method" }
  ],
  "limitations": ["未在更大规模数据上验证"],
  "futureWork": ["扩展到视觉任务"],
  "keywords": ["Transformer", "Attention"]
}
```

### 知识点 1：Section-aware 分块（按章节结构切）

**F-01/F-02 做法：** 全文 → 固定 800 字滑动窗口 → `chunk:0, chunk:1...`，块不知道属于哪一节。

**F-06 做法：** 先识别文档结构，再按语义章节切：

```
论文全文
  ├── Abstract      → section=abstract
  ├── Introduction  → section=introduction
  ├── Method        → section=method
  ├── Results       → section=results
  ├── Discussion    → section=discussion
  └── References    → section=references
```

每个 chunk 除 `content` 外还有 `section`、`sectionTitle` 等 metadata。

**MVP 识别方式：**

| 文档类型 | 识别方式 |
|----------|----------|
| Markdown | 看 `#`、`##` 标题 |
| 纯文本/PDF | 正则匹配 `Abstract`、`Introduction`、`Method` 等 |

**价值：** 贡献/方法/结论分属不同章节；RAG 可优先搜 `section=method`；每条 finding 可标注 `section` 来源。

---

### 知识点 2：信息抽取 vs 摘要

| 维度 | 摘要（F-03） | 知识抽取（F-06） |
|------|-------------|-----------------|
| 目标 | 压缩信息，快速了解大意 | 按固定字段填表，便于检索对比 |
| 输出 | 连贯段落 | 结构化 JSON |
| 信息损失 | 允许概括省略 | 尽量保留可验证事实 |
| 典型 Prompt | 「用 2～5 段概括全书」 | 「从 Method 节抽取 approach、dataset，没有就 null」 |
| 存储 | `DocumentSummary` | `DocumentKnowledge` |

- **摘要** = 读书笔记：「这本书讲了一个少年的成长故事……」
- **知识卡片** = 文献综述表：研究问题、方法、数据集、结论、局限

F-06 属于 NLP 里的 **Information Extraction（信息抽取）**，不是 Summarization。

---

### 知识点 3：分 section 独立抽取再 merge

**错误做法：** 整篇 30 页论文一次 `generateObject` → 易超 context、混节、幻觉高。

**正确做法：**

```
Abstract  → generateObject → { researchQuestion, contributions 草稿 }
Method    → generateObject → { methodology, definitions }
Results   → generateObject → { keyFindings }
Discussion→ generateObject → { limitations, futureWork }
        ↓
    merge → DocumentKnowledge.payload
```

每步仅输入该 section 文本，Prompt 约束「仅根据本节，不足则 null，禁止编造」。

Worker 流水线多一阶段：`embedding → summarizing → extracting_knowledge → ready`。

---

### 知识点 4：Structured Output（结构化输出）

`generateObject` + Zod：Schema → JSON Schema → LLM 生成 → Zod 校验 → 类型安全对象。

F-06 特殊要求：

1. **字段带溯源（provenance）**：`keyFindings[].section` 标明来自哪一节
2. **空值策略**：没写 dataset 则 `null`，不编造
3. **Schema 驱动**：不同体裁可用不同 schema（paper vs lab_report）

---

### 知识点 5：`indexMode=academic` 流水线

```
用户选 paper / lab_report
    ↓
toIndexMode() → academic
    ↓
Worker: extracting → chunking(section-aware) → embedding
      → summarizing → extracting_knowledge → ready
```

| 步骤 | general | academic |
|------|---------|----------|
| 分块 | 固定字数滑动 | 按 Abstract/Method/Results |
| 摘要 | Map-Reduce book | section 摘要 + 全书 |
| 知识抽取 | 无 | extract_knowledge |
| 前端 Tab | 摘要 | 摘要 + 知识点 |

---

### 知识点 6：RAG / 摘要 / 知识卡片分工

| 用户意图 | 走哪条线 |
|----------|----------|
| 「Table 3 准确率是多少？」 | **RAG** |
| 「这篇论文整体讲什么？」 | **Summary** |
| 「贡献和方法分别是什么？」 | **Knowledge 卡片** |
| 「和另一篇方法有何不同？」 | Knowledge 对比（F-13） |

三条线共用 `DocumentChunk`（embedding），读的路径不同：

```
DocumentChunk (embedding)  → RAG 问答
DocumentSummary (payload)  → 摘要 Tab
DocumentKnowledge (payload)→ 知识点 Tab
```

### 面试 3 分钟版

> 网盘 AI 有三条产品线。划词和 RAG 负责问答：切块、embedding、cosine Top-K 增强 Prompt。摘要负责宏观：Map-Reduce 预存全书摘要。学术知识卡片是第三条：论文 section-aware 分块，按节 `generateObject` 抽结构化字段，合并成知识卡片。用户可 RAG 问细节、看摘要把握大意、看卡片了解贡献/方法/结论，且 finding 带来源 section 减幻觉。

## 问答二：写死章节分类是否可行？

### 问题

默认文档会有「贡献」「方法」等标题，但实验报告、文献标题未必一致；文档也未必按标题分好类；每篇文献分类方式不同。写死分类提取，若某篇论文没有你定的分类，不就抽不到信息了吗？

### 结论：担心成立，不能把 section 识别写成硬前置

若实现为「必须先识别 Abstract/Method/Results 才能抽字段」，会遇到：

| 现实情况 | 问题 |
|----------|------|
| 标题是「研究方法」「实验设计」 | 英文正则失败 |
| 中文「摘要」「实验结果」 | 英文规则不适用 |
| PDF 提取后无换行、标题丢失 | 分不出 section |
| 实验报告按「目的/步骤/数据/分析」 | 与 enum 对不上 |
| 无单独 Limitations 节 | 字段永远空 |

**不能把「章节标题分类」和「知识字段 schema」绑成硬依赖。**

PRD 里的 section-aware 是 **MVP 启发式**，应视为提高准确率的优化，不是唯一路径。

---

### 关键区分：两件事不要混为一谈

**A. 知识卡片字段 schema（可相对固定）**

`contributions`、`methodology`、`keyFindings`… — 用户/UI 需要的稳定列名。

**B. 文档物理结构（高度不固定）**

`Abstract` / `摘要` / `Method` / `实验设计` — 因文而异。

正确关系：

```
固定 schema（要抽什么）
    ↑ 填充来源多样
不固定文档结构（从哪抽）
```

错误关系：

```
必须先匹配 Method 节 → 才能填 methodology  ❌
```

### 工业界常见策略（分层 + 降级）

#### 1. 模糊 / 多语言标题映射

```typescript
const SECTION_ALIASES = {
  method: ['method', 'methods', 'methodology', '研究方法', '实验方法', '实验设计'],
  results: ['results', 'findings', '实验结果', '结果与分析'],
  abstract: ['abstract', 'summary', '摘要', '内容摘要'],
}
```

归一化 + 模糊匹配，而非精确等于 `Method`。

#### 2. LLM 做结构分类

不要求原文有标题，模型根据内容判断段落属于 method / results 等。

#### 3. 无结构时全文档抽取（必须有降级）

```
识别不到 section
    ↓
退化为 general 滑动分块
    ↓
对全文或 top chunks 做一次 generateObject
    ↓
能填多少填多少，填不出 null
```

#### 4. 分字段抽取，而非死绑某一节

```
问题1：研究问题/贡献？ → abstract + introduction 相关 chunks
问题2：方法/数据集？   → 语义检索 method 相关 chunks
问题3：主要结论？     → results 相关 chunks
```

用 RAG 按语义找片段，比死盯标题更鲁棒；section 标签是加分项，不是必要条件。

#### 5. 体裁 schema 分化

`paper` 与 `lab_report` 用不同 Zod schema（F-03 已有 `summaryGenre` 先例）。

#### 6. RAG 永远兜底

卡片是预计算宏观视图；细节问题仍走 F-02 RAG。卡片抽不全 ≠ 系统失效。

### 更合理的 F-06 架构

```
                    ┌─────────────────────┐
                    │  KnowledgeCardSchema │  ← 固定（按 genre）
                    └──────────┬──────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         ▼                     ▼                     ▼
   [有清晰标题]          [标题模糊/中文]         [完全无结构]
   正则/MD 解析          LLM 分段分类           滑动窗口分块
         └─────────────────────┼─────────────────────┘
                               ▼
              按字段 RAG 检索相关 chunks（可选）
                               ▼
              分字段 generateObject
                               ▼
                    merge → DocumentKnowledge
                               ▼
                    缺失字段 = null（不编造）
```

要点：

1. Schema 固定，输入来源灵活
2. Section 是 metadata，不是 gate
3. 识别失败有降级路径
4. 按字段抽，比按节抽更抗格式差异

---

### 与 F-03 的类比

F-03 不假设每本书有章节号：有 `chapterNo` 走 chapter Reduce，没有则 chunk 直接 book。

F-06 同样应 optional：有 section 则分节抽更准；没有则全文档抽，能填多少填多少。

---

### 「如果」逐一回答

| 情况 | 做法 |
|------|------|
| 标题不一致 | 别名表 + 模糊匹配 +（可选）LLM 分类 |
| 文档没分好类 | 降级普通 chunk + 全文/分字段抽取；section=unknown |
| 每篇分类方式不同 | 不还原作者章节树，只关心哪些文本支撑哪个 schema 字段 |
| 没有某分类就抽不到 | 仅当 section 写成硬前置时才会；应 null + 其他字段照常 + RAG 兜底 |
| 分好类但标题各异 | `section` 用粗粒度 enum；`sectionTitle` 存原文标题仅供展示 |

---

### 面试进阶表述（含 trade-off）

> 学术知识卡片用固定 Zod schema 保证产品和 UI 稳定，但不假设所有论文都有 IMRaD 结构。MVP 用标题启发式提高准确率；识别失败则降级为滑动分块 + 按字段抽取；细节仍走 RAG。Section 是检索和溯源的 metadata，不是抽取的前置条件。缺失字段显式 null，避免幻觉。

---

## 实施建议（综合两次讨论）

实现 F-06 时建议：

1. **固定 schema，灵活来源** — `KnowledgeCardSchema` 按 `summaryGenre` 分化
2. **Section 识别可选** — 成功则缩小检索范围；失败走 `unknown` + 全文档路径
3. **必须实现无结构降级** — 不可假设所有 PDF/报告都有标准标题
4. **按字段抽取优于按节一一绑定** — 减少「没有 Method 节就抽不出 methodology」
5. **RAG 与摘要继续兜底** — 知识卡片是速览层，不是唯一 AI 入口
