## 问答一：F-06 会学到哪些知识点？

### 问题

> F-06 学到的知识点给我详细解释一下是什么东西。

### 背景：已掌握 vs 将要学

| 已掌握 | 来自 |
|--------|------|
| 流式 LLM、`AbortController` | F-00 |
| 离线索引、BullMQ、分块 | F-01 |
| Embedding、Cosine、Top-K RAG | F-02 |
| Map-Reduce 摘要、`generateObject` + Zod | F-03 |
| PDF 文本提取、多格式 pipeline | F-05 |

F-06（学术文献知识卡片）在你现有能力之上，会引入一条新的产品线：**线 C — 结构化知识抽取**。

下面按「是什么 → 为什么 → 怎么做 → 和已做功能的区别 → 面试怎么说 → 实现时会碰什么」展开。

---

### 一、F-06 到底在做什么？

你已经有的：

| 功能 | 回答的问题 |
|------|-----------|
| F-02 RAG | 「论文里某句话在哪？某细节是什么？」 |
| F-03 摘要 | 「这篇论文/小说整体讲什么？」 |

F-06 回答的是第三个问题：

> **「这篇论文有哪些结构化知识点？贡献是什么？方法是什么？关键发现有哪些？局限在哪？」**

输出不是一段自由文本，而是一张**固定字段的知识卡片 JSON**，前端用卡片 UI 展示。

示例结构（设计文档 schema，见路线图 §5.1）：

```json
{
  "title": "Attention Is All You Need",
  "researchQuestion": "能否只用注意力机制做序列建模？",
  "contributions": ["提出 Transformer", "..."],
  "methodology": {
    "approach": "Self-Attention",
    "dataset": "WMT 英德翻译",
    "metrics": ["BLEU"]
  },
  "keyFindings": [
    {
      "claim": "Transformer 在翻译任务上优于 RNN",
      "evidence": "Table 2",
      "section": "results"
    }
  ],
  "definitions": [
    { "term": "Multi-Head Attention", "definition": "...", "section": "method" }
  ],
  "limitations": ["未在更大规模数据上验证"],
  "futureWork": ["扩展到视觉任务"],
  "keywords": ["Transformer", "Attention"]
}
```

存进 `DocumentKnowledge` 表（`payload` 为 JSON），用户打开「知识点」Tab **直接读库**，不实时跑 LLM。

PRD 验收标准强调：

- mock 论文 JSON 含 `contributions`、`keyFindings`（带 `section`）
- **无编造字段**（空则 `null`）

### 二、核心知识点 1：Section-aware 分块（按章节结构切）

#### 你现在 F-01/F-02 的做法

```
全文 → 固定 800 字一块、overlap 100 → chunk:0, chunk:1, chunk:2...
```

每块**不知道**自己属于 Abstract 还是 Method，只是按字数滑动窗口切。

#### F-06 的做法

先识别文档结构，再按**语义章节**切：

```
论文全文
  ├── Abstract      → section=abstract
  ├── Introduction  → section=introduction
  ├── Method        → section=method
  ├── Results       → section=results
  ├── Discussion    → section=discussion
  └── References    → section=references
```

每个 chunk 除了 `content`，还有 metadata（与 Prisma `DocumentChunk.section` 对齐）：

```typescript
{
  chunkIndex: 3,
  section: 'method',        // DocumentChunkSection 枚举
  sectionTitle: 'Method',   // 原文标题，仅供展示
  content: '...'
}
```

数据库里 `DocumentChunkSection` 已是粗粒度枚举（不是复刻作者 50 种标题）：

```
unknown | abstract | introduction | method | results | discussion | references | body | chapter
```

#### 怎么识别 section（MVP）

| 文档类型 | 识别方式 |
|----------|----------|
| Markdown | 看 `#`、`##` 标题 |
| 纯文本/PDF 提取文本 | 正则匹配 `Abstract`、`Introduction`、`Method`、`Results` 等 |

> 注意：这只是 MVP 启发式，**不是唯一路径**。识别失败时的降级策略见 [问答二](#问答二写死章节分类是否可行)。

#### 为什么要这样切？

1. **贡献、方法、结论本来就在不同章节** — 按 section 切，抽「方法」时不会混进「结论」
2. **RAG 更准** — 问「实验用了什么数据集？」可以优先搜 `section=method` 或 `section=results`
3. **每条 finding 能标注来源** — `keyFindings[].section = "results"`，知道结论来自哪一节

这是**文档结构感知（structure-aware）**，比纯滑动窗口分块更贴近真实阅读场景。

### 三、核心知识点 2：信息抽取 vs 摘要（两种 LLM 任务）

很多人会混，面试也常问。

| 维度 | 摘要（F-03） | 知识抽取（F-06） |
|------|-------------|-----------------|
| 目标 | 压缩信息，让人快速了解大意 | 按固定字段**填表**，便于检索和对比 |
| 输出 | 连贯段落（`overview`、`plotPoints`） | 结构化 JSON（`contributions`、`methodology`） |
| 信息损失 | 允许概括、省略细节 | 要尽量保留可验证的事实（数据集、指标、Table 编号） |
| 典型 Prompt | 「用 2～5 段概括全书」 | 「从 Method 节抽取 approach、dataset、metrics，没有就 null」 |
| 存储 | `DocumentSummary` | `DocumentKnowledge` |

类比：

- **摘要** = 写读书笔记：「这本书讲了一个少年的成长故事……」
- **知识卡片** = 填文献综述表：研究问题、方法、数据集、主要结论、局限

F-06 属于 NLP 里的 **Information Extraction（信息抽取）**，不是 Summarization。

---

### 四、核心知识点 3：分 section 独立抽取再 merge

#### 错误做法（一次抽全书）

```
整篇论文 30 页 → 一次 generateObject → 一张大 JSON
```

问题：

- 超出 context 窗口
- 模型容易把 Method 和 Results 混在一起
- 幻觉风险高（没写的内容也会被「编」进 `contributions`）

#### F-06 正确做法（理想路径：有清晰 section 时）

```
Abstract 文本    → generateObject → { researchQuestion, contributions 草稿 }
Method 文本      → generateObject → { methodology, definitions }
Results 文本     → generateObject → { keyFindings }
Discussion 文本  → generateObject → { limitations, futureWork }
        ↓
    merge 合并成一张 DocumentKnowledge.payload
```

每一步：

- **输入**：仅该 section 的文本（不跨节）
- **Schema**：该 section 负责的字段子集
- **约束**：Prompt 写「仅根据本节，不足则 `null`，禁止编造」

这叫 **divide-and-conquer extraction（分而治之抽取）**，和 F-03 Map-Reduce 摘要思路类似，但目标是填结构化字段，不是写段落。

> **定稿说明（见 [抽取方案定稿](#抽取方案定稿已确认)）**：按节抽取是 PRD/路线图描述的「理想路径」。**实现上以分字段 RAG + `generateObject` 为主引擎**；section 识别用于缩小检索范围和写 provenance，不是硬前置。有结构时可在字段组检索中 boost 对应 section，而非「一节一抽」绑定。

Worker 里会多一个状态（`DocumentIndexStatus` 已预留）：

```
embedding → summarizing → extracting_knowledge → ready
                              ↑ F-06 新增
```

路线图 §7.3 还约定：Section 摘要可并行写入 `DocumentSummary`（`type=section`），与知识卡片抽取独立。

### 五、核心知识点 4：Structured Output（结构化输出）

F-03 你已经用过 `generateObject` + Zod，F-06 会用到更深一层。

#### 机制（复习 + 加深）

```
Zod Schema  →  转成 JSON Schema  →  发给 LLM
                    ↓
              LLM 按字段生成 JSON
                    ↓
              Zod 校验 parse
                    ↓
              类型安全的 TypeScript 对象
```

#### F-06 的特殊要求

**1. 字段要带 provenance（溯源）**

```typescript
keyFindings: Array<{
  claim: string;      // 发现了什么
  evidence: string;   // 依据（Table 2 / Fig.1）
  section: string;    // 来自哪一节 ← 重要
}>
```

面试可以说：结构化输出不只是「格式对」，还要「每条事实能追溯到文档结构」。

**2. 空值策略**

- 论文没写 dataset → `dataset: null`，不是瞎编 `"ImageNet"`
- 和 F-03 摘要里「不足则 `[]`」是同一类防幻觉思路

**3. Schema 设计是产品能力**

不同文档类型用不同 schema：

- 论文 → `contributions`、`methodology`、`keyFindings`
- 教材 → 可能是 `definitions`、`theorems`（远期）

这叫 **schema-driven AI**，比「随便问 LLM 一句话」更可控、更可测。

### 六、核心知识点 5：`indexMode=academic` 流水线

F-06 不是单独功能，而是 **`academic` 索引模式**下的一条支路。

#### 两个概念不要混

| 字段 | 粒度 | 用途 |
|------|------|------|
| `summaryGenre` | 细（6 种：`novel` / `paper` / `lab_report` …） | 决定用什么 Prompt、什么 Zod schema 生成**摘要** |
| `mode`（`DocumentIndexMode`） | 粗（2 种：`general` / `academic`） | 决定索引流水线的处理方式（是否跑知识抽取） |

映射（S13 已实现 `toIndexMode()`）：

| `summaryGenre` | → `mode` |
|----------------|----------|
| `novel`、`general_nonfiction`、`technical`、`textbook` | `general` |
| `lab_report`、`paper` | `academic` |

#### 完整 Worker 流水线（academic）

```
用户选体裁 lab_report / paper
    ↓
toIndexMode() → mode = 'academic'
    ↓
Worker:
  1. extracting        提取 PDF/文本/DOCX
  2. chunking          section-aware 分块（不是纯 800 字滑动）
  3. embedding         向量化（和 F-02 一样，但 chunk 带 section 标签）
  4. summarizing       Map-Reduce 摘要（section 摘要 + book 摘要）
  5. extracting_knowledge  ← F-06 核心
     分 section / 分字段抽取 → merge → 写 DocumentKnowledge
  6. ready
```

和 `general` 模式对比：

| 步骤 | general（小说/技术文档） | academic（论文/实验报告） |
|------|------------------------|--------------------------|
| 分块 | 固定字数滑动窗口 | 优先 section-aware；失败则降级滑动窗口 |
| 摘要 | Map-Reduce → book 摘要 | section 摘要 + 全书摘要 |
| 知识抽取 | 无 | **extract_knowledge → 知识卡片** |
| 前端 Tab | 摘要 | 摘要 + **知识点** |

`summaryGenre`（`paper` / `lab_report`）决定摘要 schema 和知识卡片 schema；`indexMode=academic` 决定是否跑知识抽取。两者配合，但职责不同。

### 七、核心知识点 6：RAG / 摘要 / 知识卡片 怎么分工

设计文档 §7.4 决策表，面试可以直接背：

| 用户意图 | 走哪条线 |
|----------|----------|
| 「Table 3 的准确率是多少？」 | **RAG**（检索 Results 节 chunk） |
| 「这篇论文整体讲什么？」 | **Summary**（读 book 摘要） |
| 「贡献和方法分别是什么？」 | **Knowledge 卡片**（读 `DocumentKnowledge`） |
| 「和另一篇论文方法有何不同？」 | Knowledge 卡片对比（F-13 延伸） |
| 主题定调 + 需要原文例证 | Summary 定调 + RAG 找片段 |

三条线共用同一套 `DocumentChunk`（embedding），但**读的路径不同**：

```
DocumentChunk (embedding)   ──→  RAG 问答
DocumentSummary (payload)   ──→  摘要 Tab
DocumentKnowledge (payload) ──→  知识点 Tab
```

这是**一套索引，多种 AI 产品形态**，比「每个功能各建一套 embedding」更工程化。

### 九、面试可以怎么讲（3 分钟版）

> 我的网盘 AI 有三条产品线。  
> 划词和 RAG 负责问答：把文档切块、做 embedding，用 cosine 找 Top-K 片段增强 Prompt。  
> 摘要负责宏观理解：Map-Reduce 生成全书摘要，预存库，避免每次扫原文。  
> 学术知识卡片是第三条线：针对论文，先做 section-aware 分块，识别 Abstract、Method、Results，再按节用 `generateObject` 抽取结构化字段，合并成一张知识卡片。  
> 这样用户既可以 RAG 问细节，也可以看摘要把握大意，还可以看卡片快速了解贡献、方法和结论，且每条 finding 带来源 section，减少幻觉。

## 问答二：写死章节分类是否可行？

### 问题（原文）

> 你现在的做法是默认了文档会有这些类似于「贡献」、「方法」的标题，但是如果文档里面不是这种标题呢？就算是实验报告、文献，标题也未必一致，或者文档也未必按标题给你分好类，就算真的给 AI 输入分好类的文献，每篇文献的分类方式也很可能不同，你怎么能写死分类来提取呢？如果某篇论文没有你定下来的分类，你不就提取不到信息了吗？

你的质疑是对的，而且这正是 F-06 设计里**最需要想清楚的点**。

下面分开说：**问题是否成立**、**当前方案实际在做什么**、**工业界怎么处理**、**推荐架构**。

---

### 1. 你的担心成立

如果实现成：

> 「必须先识别出 Abstract / Method / Results，再按写死的节去抽字段」

那确实会遇到这些问题：

| 现实情况 | 写死分类的问题 |
|----------|----------------|
| 标题是「研究方法」「实验设计」而不是 Method | 正则匹配失败 |
| 中文论文用「摘要」「引言」「实验结果」 | 英文规则不适用 |
| PDF 提取后没有换行、标题丢失 | 根本分不出 section |
| 实验报告按「实验目的 / 步骤 / 数据 / 分析」写 | 和你的 enum 对不上 |
| 综述、技术报告、学位论文结构各异 | 同一套 section 套不住 |
| 某篇论文没有单独的 Limitations 节 | `limitations` 字段永远空 |

所以：**不能把「章节标题分类」和「知识字段 schema」绑死成一条硬依赖**。否则覆盖率会很差。

---

### 2. 当前 PRD 里的 F-06 实际在做什么？

设计文档 §7.3 写的是 **MVP 启发式**，不是「全世界论文都必须长成一种样子」：

```
Section 识别（MVP）：
  - Markdown：看 # / ##
  - 纯文本：正则匹配 Abstract、Introduction、Method 等
```

也就是说，团队心里默认的是：

1. **Section 识别是辅助手段**，用来提高准确率，不是唯一入口
2. **最终产物是 `KnowledgeCardSchema` 那张表**（`contributions`、`methodology`、`keyFindings`…），不是「还原论文章节树」
3. **字段缺失用 `null` / `[]`**，不是硬凑
4. 还有 **RAG（F-02）** 兜底问细节

但 PRD **没把「识别失败怎么办」写得很细**——你的质疑正好补上了这块。实现时必须显式做降级，否则会掉进「没有 Method 标题就什么都抽不出来」的坑。

---

### 3. 关键区分：两件事不要混为一谈

#### A. 知识卡片的字段 schema（相对固定）

```typescript
contributions, methodology, keyFindings, limitations...
```

这是 **「用户想看到什么信息」**，类似文献综述表的列名。  
面试、产品、UI 需要稳定字段，所以 schema 可以相对固定。

#### B. 文档的物理结构（高度不固定）

```
Abstract / 摘要 / 内容摘要
Method / 方法 / 实验设计 / 3. Methodology
```

这是 **「原文怎么排版」**，因文而异，不能写死。

正确关系应该是：

```
固定 schema（要抽什么）
    ↑ 填充来源可以是多样的
不固定的文档结构（从哪抽）
```

而不是：

```
必须先匹配到 Method 节 → 才能填 methodology  ❌ 太脆
```

---

### 4. 工业界怎么处理「结构不统一」？

常见是 **分层 + 降级**，不是一条路走到黑。

#### 策略 1：模糊 / 多语言标题映射（MVP 可做）

不是只认 `Method`，而是维护 **同义词表**：

```typescript
const SECTION_ALIASES = {
  method: ['method', 'methods', 'methodology', '研究方法', '实验方法', '实验设计'],
  results: ['results', 'result', 'findings', '实验结果', '结果与分析'],
  abstract: ['abstract', 'summary', '摘要', '内容摘要'],
  introduction: ['introduction', '引言', '背景'],
  discussion: ['discussion', '讨论', '结论与讨论'],
  limitations: ['limitations', 'limitation', '局限', '不足'],
  // ...
}
```

匹配时用 **归一化 + 模糊匹配**（包含关键词、编辑距离），而不是精确等于 `Method`。

#### 策略 2：LLM 做结构分类（比正则稳）

把全文或按固定窗口切成几段，让模型输出：

```json
{
  "segments": [
    { "label": "method", "start": 1200, "end": 4500, "confidence": 0.85 },
    { "label": "results", "start": 4500, "end": 7200, "confidence": 0.9 }
  ]
}
```

**不要求原文有标题**，模型根据内容判断「这段像在讲方法还是结果」。  
成本高一些，但对 PDF、中文、格式乱的文档更稳。

#### 策略 3：无结构时的全文档抽取（必须有降级）

Section 识别失败时：

```
识别不到 section
    ↓
退化为现在的 general 分块（800 字滑动窗口）
    ↓
对全文（或 top 相关 chunks）做一次 generateObject
    ↓
能填多少填多少，填不出就 null
```

这样不会出现「没有 Method 标题就什么都抽不出来」。

#### 策略 4：分字段抽取，而不是分 section 抽取

不必「Method 节 → methodology 字段」一一对应，可以：

```
问题1：这篇论文的研究问题/贡献是什么？  → 输入：abstract + introduction 相关 chunks
问题2：用了什么方法/数据集？          → 输入：全文里 method 相关 chunks（RAG 检索 "method dataset"）
问题3：主要结论？                    → 输入：results 相关 chunks
```

用 **RAG 按语义找相关片段**，比死盯章节标题更鲁棒。  
Section 标签只是 **加分项**（缩小检索范围），不是 **必要条件**。

#### 策略 5：体裁 schema 分化（你们已在 F-03 做过）

- `paper` 和 `lab_report` 用不同 Zod schema
- 实验报告可以是：`objective`、`procedure`、`data`、`analysis`、`conclusion`
- 期刊论文才是：`contributions`、`methodology`、`keyFindings`

**不是一套分类打天下**，而是 `summaryGenre` 决定「填哪张表」。

#### 策略 6：RAG 永远兜底

知识卡片是 **预计算的宏观视图**；  
用户问「第三组实验的 p 值是多少」这种细节，仍走 **F-02 RAG**。

卡片抽不全 ≠ 系统废了，只是「速览层」不完整。

---

### 5. 更合理的 F-06 架构（建议按这个理解/实现）

```
                    ┌─────────────────────┐
                    │  KnowledgeCardSchema │  ← 固定（按 genre）
                    │  (要抽哪些字段)      │
                    └──────────┬──────────┘
                               │
         ┌─────────────────────┼─────────────────────┐
         ▼                     ▼                     ▼
   [有清晰标题]          [标题模糊/中文]         [完全无结构]
   正则/MD 解析          LLM 分段分类           滑动窗口分块
         │                     │                     │
         └─────────────────────┼─────────────────────┘
                               ▼
              按字段 RAG 检索相关 chunks（可选）
                               ▼
              分字段 generateObject（不是死绑某一节）
                               ▼
                    merge → DocumentKnowledge
                               ▼
                    缺失字段 = null（不编造）
```

要点：

1. **Schema 固定，输入来源灵活**
2. **Section 是 metadata，不是 gate**
3. **识别失败有降级路径**
4. **按字段抽，比按节抽更抗格式差异**

---

### 6. 直接回答你的几个「如果」

#### 「标题不一致怎么办？」

用别名表 + 模糊匹配 +（可选）LLM 分类，不要只认英文 `Method`。

#### 「文档根本没分好类怎么办？」

降级为普通 chunk + 全文/分字段抽取；`section` 字段标 `unknown`，照样尝试填 schema。

#### 「每篇文献分类方式不同怎么办？」

不追求还原作者的章节树；只关心 **哪些文本片段能支撑哪个 schema 字段**。  
作者的「3.2 实验环境」和你的 `method` 标签可以不一致，只要内容能被归到 `methodology` 即可。

#### 「某篇论文没有你定的分类，就抽不到？」

**只有在你把 section 识别写成硬前置时才会这样**——这是实现 bug，不是方案必然。  
正确做法：该字段 `null`，其他字段照常抽；用户仍可用 RAG 问。

#### 「就算分好类，每篇分类方式也不同？」

所以 `DocumentChunk.section` 应是 **粗粒度枚举**（`method` / `results` / `body` / `unknown`），  
不是复刻作者原文的 50 种标题；`sectionTitle` 存原文标题仅供展示。

---

### 7. 和 F-03 的类比（帮你建立直觉）

F-03 摘要也没有假设「每本书都有章节号」：

- 有 `chapterNo` → 走 chapter Reduce
- 没有 → 跳过，chunk 直接 Reduce 成 book

F-06 应同样 **optional enhancement**：

- 有 section → 检索范围更准、provenance 更好写
- 没有 → 仍走分字段 RAG 抽取，能填多少填多少

---

### 8. 面试时怎么说（显得你思考过 trade-off）

> 学术知识卡片用固定 Zod schema 保证产品和 UI 稳定，但不假设所有论文都有 IMRaD 结构。MVP 用标题启发式提高准确率；识别失败则降级为滑动分块 + 按字段抽取；细节问题仍走 RAG。Section 是检索和溯源的 metadata，不是抽取的前置条件。缺失字段显式 null，避免幻觉。

这比只说「我们按 Abstract/Method/Results 抽」要成熟得多。

---

### 问答二 · 一句话总结

你的质疑没错：**不能写死「必须先有 Method 节才能抽方法」**。

合理做法是：

- **固定的是「要抽什么字段」（schema）**
- **不固定的是「从哪段原文抽」（结构识别 + 降级 + RAG）**
- **抽不到就 null，别编造；细节交给 RAG**

PRD 里的 section-aware 分块是 **提高准确率的优化**，不是 **唯一路径**。实现 F-06 时务必做 **无结构降级**，否则就会掉进你说的坑。

## 抽取方案定稿（已确认）

> 2026-07-12 确认：六种策略**组合使用**，不是六选一。抽取主引擎统一为**分字段 RAG + `generateObject`**。

### 结论一句话

**MVP 组合：策略 1 + 3 + 4 + 5（策略 6 沿用现有 RAG）；策略 2（LLM 结构分类）后续按需加。Section 识别只辅助检索和溯源，不做抽取硬门槛。**

---

### 六种策略各自管什么

| 策略 | 解决什么问题 | 在流水线里的角色 | MVP |
|------|-------------|-----------------|-----|
| **1. 别名表 / 模糊匹配** | 文档有没有章节结构？ | 结构识别（便宜） | ✅ 做 |
| **2. LLM 结构分类** | 标题不规范时怎么分段？ | 结构识别（增强，成本高） | ⏸ 暂缓 |
| **3. 无结构降级** | 识别失败怎么办？ | 安全网（必须有） | ✅ 做 |
| **4. 分字段 RAG 抽取** | 怎么填 schema 每个字段？ | **抽取引擎（核心）** | ✅ 做 |
| **5. 体裁 schema 分化** | `paper` / `lab_report` 填哪张表？ | 产品层（始终开启） | ✅ 做 |
| **6. RAG 问答兜底** | 卡片抽不全用户怎么办？ | 用户入口（F-02 已有） | ✅ 已有 |

三层分工：

```
┌─────────────────────────────────────────┐
│  第 1 层：结构识别（策略 1 → 2 → 3）      │  尽量知道「哪段像 method/results」
├─────────────────────────────────────────┤
│  第 2 层：字段抽取（策略 4 + 5）          │  真正往 KnowledgeCard 里填值
├─────────────────────────────────────────┤
│  第 3 层：产品兜底（策略 6）              │  用户问细节时走 RAG，不依赖卡片
└─────────────────────────────────────────┘
```

策略 **5、6** 与 1～4 不竞争：5 决定填什么表，6 是用户侧兜底。

---

### MVP 流水线（定稿）

```
文本
  ↓
[策略 1] 尝试 section 识别（MD 标题 + 别名表）
  ↓ 成功                              ↓ 失败
给 chunk 打 section 标签           [策略 3] 走现有 chunkText 滑动分块，section=unknown
  ↓                                        ↓
              embedding（复用 F-01/F-02 已有能力）
                          ↓
              [策略 4] 按字段 RAG 抽取（核心，两条路径共用）
                - 每个字段（或字段组）检索 top-K chunks
                - generateObject 填该字段子集
                - 有 section 标签时优先 filter/加权，没有也能跑
                          ↓
              [策略 5] 按 summaryGenre 选 schema（paper / lab_report）
                          ↓
              merge → DocumentKnowledge（缺失字段 = null）
                          ↓
用户看知识点 Tab；细节问不到 → [策略 6] 已有 RAG 问答
```

**关键原则：**

- **策略 4 是抽取主引擎**，不做成「必须先有 Method 节才能抽 methodology」
- **策略 1 是加分项**：有结构时检索更准、`keyFindings[].section` 更好写
- **策略 3 是保底**：`section-parser` 失败不阻塞整条流水线
- **策略 2 以后加**：当别名表覆盖不够、有真实 bad case 时再上 LLM 分段

### 抽取主方法：分字段 RAG + `generateObject`（策略 4）

不按「一节一抽」绑定，而是**按 schema 字段组**分别检索 + 生成：

| 字段组 | 检索 query（示例） | 有 section 时优先过滤 |
|--------|-------------------|----------------------|
| `researchQuestion` + `contributions` | `"research question contribution"` | `abstract` / `introduction` |
| `methodology` | `"method dataset metric experiment"` | `method` |
| `keyFindings` | `"result finding experiment outcome"` | `results` |
| `limitations` + `futureWork` | `"limitation future work"` | `discussion` |
| `definitions` | `"definition term concept"` | 全文（或 `method` + `introduction`） |
| `title` + `keywords` | `"title keywords"` | `abstract` + 文首 chunks |

每组调用伪代码：

```typescript
async function extractFieldGroup(
  fieldGroup: FieldGroup,
  chunks: EmbeddedChunk[],
  schema: ZodSchema,
  options?: { preferSections?: DocumentChunkSection[] },
): Promise<Partial<KnowledgeCard>> {
  const candidates = options?.preferSections?.length
    ? rankChunks(chunks, fieldGroup.query, { boostSections: options.preferSections })
    : rankChunks(chunks, fieldGroup.query);

  const topK = candidates.slice(0, FIELD_GROUP_TOP_K);

  return generateObject({
    schema: fieldGroup.partialSchema,
    prompt: buildFieldPrompt(fieldGroup, topK),
    // Prompt 约束：仅根据以下片段；不足则 null；禁止编造
  });
}
```

合并：各字段组 partial result → `mergeKnowledgePayload()` → 完整 `DocumentKnowledge.payload`。

### 为什么不建议「只选一种策略」

| 如果只选… | 后果 |
|----------|------|
| 只做 1（别名表） | 中文 / 乱 PDF 一挂就全挂 |
| 只做 2（LLM 分段） | 贵，且分段 ≠ 字段填对 |
| 只做 3（全文抽） | 能跑，但长文易混节、幻觉高 |
| 只做 4（分字段 RAG） | 可单独跑，但缺 section 标签时溯源和精度差一截 |
| 只做 5 | 只有 schema，没有抽取逻辑 |
| 只做 6 | 只有问答，没有预计算知识卡片 |

**最小可靠组合：1（尝试）+ 3（失败兜底）+ 4（抽取）+ 5（schema）。**

### 路径 A / 路径 B 与抽取方案的关系

两条路径是**同一条流水线的分块分支**，共享同一套抽取引擎（策略 4），**不是**两套互斥方案：

| | 路径 A（有结构） | 路径 B（无结构降级） |
|--|----------------|---------------------|
| 触发条件 | `section-parser` 成功且 confidence ≥ 阈值 | 识别失败或 confidence 低 |
| 分块 | section-aware，写入 `DocumentChunk.section` | 现有 `chunkText` 滑动窗口，`section=unknown` |
| **抽取** | **策略 4 分字段 RAG** | **策略 4 分字段 RAG** |
| 差异 | 检索可 filter/boost by section；provenance 更准 | 纯语义检索；`section` 标 `unknown` |

> ❌ 已否决：「有结构 → 按节 `generateObject`；无结构 → 才分字段 RAG」——路径 A 太脆，与问答二的结论矛盾。

路径选择伪代码（定稿）：

```typescript
const parseResult = await parseSections(text);

const chunks =
  parseResult.confidence >= SECTION_CONFIDENCE_THRESHOLD &&
  parseResult.segments.length >= 2
    ? chunkBySections(text, parseResult)   // 路径 A：分块带标签
    : chunkText(text).map((c) => ({ ...c, section: 'unknown' })); // 路径 B

await embedChunks(chunks);

// 两条路径汇合：统一走分字段 RAG 抽取
return extractKnowledgeByFieldRag(chunks, pickKnowledgeSchema(summaryGenre));
```

### 五条铁律（Code Review 检查项）

1. **固定 schema，灵活来源** — `KnowledgeCardSchema` 按 `summaryGenre` 分化
2. **Section 识别可选** — 成功则缩小检索范围；失败走 `unknown` + 全文档路径
3. **必须实现无结构降级** — 不可假设所有 PDF/报告都有标准标题
4. **按字段抽取是主引擎** — 不按「Method 节存在与否」决定能否抽 `methodology`
5. **RAG 与摘要继续兜底** — 知识卡片是速览层，不是唯一 AI 入口
