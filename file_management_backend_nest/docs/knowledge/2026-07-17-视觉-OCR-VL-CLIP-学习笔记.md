## 问答一：截图提取信息分两类，分别用什么？

### 问题

> 截图提取信息要怎么做，这里分 2 类：  
> （1）图片里面是纯文字，我用什么技术来提取图片中的文字。  
> （2）如果图片不是文字，比如是猫狗的图片，然后假设我让 AI 帮我给这些图片分类整理，这个要怎么做？

### 总览

这两类问题看起来都是「从图片里拿信息」，但底层技术完全不同：

- **（1）图里是字** → 要的是「把图变成可搜索的正文」→ 用 **OCR**。
- **（2）图里是物体/场景（猫狗）** → 要的是「理解图里有什么并归类」→ 用 **视觉理解（VL）或图像分类（含 CLIP）**，不是 OCR。

为什么要分开？因为 OCR 只负责「认字」；猫狗图上通常没有可检索的成段文字，强行走 OCR 只会得到空结果或噪声。分类整理需要的是「看懂画面」，属于另一条能力线。

---

### （1）图片里是纯文字：用什么技术提字？

**推荐技术：OCR（Optical Character Recognition，光学字符识别）。**

OCR 的任务很明确：输入一张图片，输出图上的文字内容（以及可选的文字框位置、置信度等）。对「课件截图、聊天记录截图、扫描页、白板拍照」这类以字为主的图，OCR 是正路。

**常见实现选择（由轻到重）：**

1. **开源本地 OCR**  
   - 如 Tesseract、PaddleOCR。  
   - 优点：可控、可离线、无按次费用。  
   - 缺点：中文/复杂版式/手写效果参差；要自己做部署与运维。

2. **云端 OCR API**  
   - 如各家云厂商 OCR、或面向文档的 OCR 服务。  
   - 优点：准确率通常更好，省自建模型。  
   - 缺点：有调用成本与隐私外传问题。

3. **文档/版面 OCR（Document AI）**  
   - 不仅认字，还尽量保留段落、表格结构。  
   - 适合「整页文档截图」而不只是零散一行字。

4. **用多模态大模型（VL）直接读图出字**  
   - 把图喂给 VL，让它「把图上的字原样打出来」。  
   - 优点：对复杂版式、图文混排有时更懂上下文。  
   - 缺点：成本高、延迟大、可能「脑补」文字；**大批量建索引一般不如专用 OCR 划算**。

**对本项目（F-26）的建议：**  
索引阶段用 **专用 OCR（如 DeepSeek-OCR）** 抽文本，再进入现有 `chunk → embed → 存 DocumentChunk` 流水线；不要把「批量建索引」绑在贵的交互型 VL 上。

## 问答二：OCR 是否只是现有向量管线的前置步骤？

### 问题（用户原话整理）

> 我理解的 OCR 大概是抽出来文字信息之后，让这些文字信息走现有的向量数据库匹配这一套，本质上只是多了一个前置步骤，怎么从图片里面提取文字的事情。然后后面文字转向量可以复用现在的一整套功能。

### 详细回答

**是的，可以这么理解。**

当前文本文件的索引流水线大致是：

```
文件字节 ──文本提取──► 纯文本 ──切块──► chunks ──embedOne──► 向量入库 ──► 检索/RAG
```

图片（文字型）接入后变成：

```
图片字节 ──OCR──► 纯文本 ──切块──► chunks ──embedOne──► 向量入库 ──► 检索/RAG
```

也就是说：

- **变的是「文本从哪来」**：以前从 txt/pdf/docx 解析；现在从图片 OCR。  
- **不变的是「文本之后做什么」**：切块规则、embedding 模型、存 `DocumentChunk`、余弦相似度检索、按文件聚合（F-08）都可以复用。

因此 F-26 的核心新增点是 `extractTextFromImage`（以及图片 MIME 准入），而不是重做一套向量库。

**边界提醒：**

- 若 OCR 结果为空或过短，应标记索引失败/不可用，而不是把空文本当真内容去 embed。  
- OCR 质量（错字、漏行、版面打乱）会直接传导到检索质量；前置步骤准，后面整条链才准。

## 问答三：VL 与 CLIP 的原理是什么？

### 问题

> 但是 VL 或 CLIP 是什么原理，介绍一下。

### VL：多模态大模型（看图 + 按指令生成）

**VL** = Vision-Language Model，视觉语言模型。它能同时接收图像和文本，并按你的问题生成回答。

**工作原理（简化）：**

1. 图片被视觉编码器转成一组向量/token（模型「看见」画面里的物体、布局、文字区域等）。  
2. 这些视觉信息与用户的文字问题一起进入语言模型。  
3. 模型按指令生成输出：描述、分类标签、解题步骤、是否含敏感信息等。

**和 OCR 的关键差别：**

- OCR 的任务几乎固定：「图上的字是什么」。  
- VL 的任务由 prompt 决定：「这是什么」「帮我归类」「这道题怎么解」「有没有证件信息」……  
- 对猫狗图：应走 VL（或分类模型），而不是 OCR。

**成本与架构提示：**  
VL 调用通常比 OCR 更贵。项目约定把「索引用 OCR」和「交互用 VL」分开，避免所有图片一上传就打昂贵的 VL。

### CLIP：图文对比学习（同一向量空间）

**CLIP** 把图片和文本编码到**同一个向量空间**，用相似度判断「这张图」和「这句话」是否匹配。

**工作原理（简化）：**

1. 训练阶段：大量「图 + 对应说明」正样本，以及不匹配的负样本。  
2. 学习目标：配对的图文向量靠近，不配对的远离（contrastive learning）。  
3. 使用阶段：  
   - 图像 → 图像向量  
   - 文本 → 文本向量  
   - 两者做余弦相似度

**能做什么：**

- **零样本分类**：不专门训「猫狗分类器」，也能用「a photo of a cat / dog」这类提示做分类。  
- **以文搜图**：搜「海边的狗」，用文本向量在图像向量库里找近邻。  
- **以图搜图**：用一张图找相似图。

**和 VL 的差别：**

- CLIP：偏 **检索 / 匹配 / 固定类目打分**，输出主要是向量与相似度。  
- VL：偏 **开放式生成**（任意描述、任意分类体系、解题步骤）。

**和现有文本 embedding 的差别：**

- 现有 `embedOne`：进的是 **文本 chunk**，空间是「文本语义空间」。  
- CLIP 图像向量：进的是 **图文对齐空间**，与现有文本库通常不通用。  
- 若希望「用现在的文本语义搜也能搜到猫图」，更常见的工程做法是：VL 生成 caption/标签 → 再对这段文字做 `embedOne` → 复用现有文本检索（代价是多一次 VL 调用）。

### 三条路径总览

```
（1）OCR：图 → 字 → 现有文本向量管线（F-26）
（2）VL：图 + 指令 → 生成标签/描述/答案（F-31/F-27）
（3）CLIP：图/文 → 同一空间向量 → 相似度检索或分类
```

## 和本项目路线的对应

| 能力 | PRD | 技术侧重点 |
|------|-----|------------|
| 图片可索引 | **F-26** | OCR → 复用 chunk/embed/RAG |
| 扫描 PDF | **F-30** | 页渲染 + OCR |
| 图片自动打标签 | **F-31** | OCR 关键词 和/或 轻量 VL |
| 截图解题 | **F-27** | 交互 VL；不进 index Worker |
| 语义搜文件 | **F-08** | 文本向量；OCR 入库后截图可被搜到 |

**架构约束：** 索引用 OCR、交互用 VL，分开配置与计费；不要把所有视觉能力塞进 `document-index.processor`。
