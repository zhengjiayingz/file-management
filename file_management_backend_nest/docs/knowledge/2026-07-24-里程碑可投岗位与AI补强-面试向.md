## 1. 结论：怎么定位自己

当前网盘 + Document Intelligence 里程碑的**最强卖点**是：

> 能把 AI 嵌进完整业务系统（鉴权、文件、异步任务、前端可演示），而不是调过几个大模型 API。

**简历一句话定位建议：**

> NestJS 全栈 / AI 应用方向：自研网盘上落地 RAG、多模态索引、ASR/TTS 与异步任务编排，可演示、可回归测试。

**不要写进主定位的：** 算法工程师、NLP 研究员。

## 2. 当前证据链（项目能证明什么）

### 2.1 业务后端 / 全栈

| 能力 | 技术 |
|------|------|
| 后端框架 | NestJS + TypeScript（Express → Nest 迁移完成） |
| 前端 | Vue 3 + TypeScript |
| 数据与存储 | MySQL + Prisma；MinIO / 本地对象存储 |
| 异步与实时 | BullMQ Worker、Redis、Socket.IO |
| 业务面 | 上传、预览、版本、分享、VIP、查重、管理端等 |

### 2.2 AI 应用（Document Intelligence）

| 能力线 | 代表能力 |
|--------|----------|
| 文本智能 | 划词流式问答、RAG、Embedding、语义搜索、分层摘要、知识卡片 / 结构化抽取 |
| 多模态 | 扫描 PDF / OCR、VL 解题、以图搜图、错题本 |
| 音视频 | ASR（音频转写 + segments）、TTS；视频抽音轨为下一里程碑 |
| Agent 入门 | AI 助手只读 Function Calling MVP |
| 工程 | Provider 抽象、`ai/` 按应用域分层、Jest 回归 |

**明确非目标（路线图已写）：** 自训模型 / Fine-tuning；本阶段不主打独立向量集群（MVP 用 MySQL JSON + 应用层相似度）。

## 3. 可投岗位（按匹配度）

| 优先级 | 岗位名（JD 常见写法） | 为什么能投 | 面试主线怎么讲 |
|--------|----------------------|------------|----------------|
| **主投** | Node.js / NestJS 后端开发 | Nest 迁移、鉴权、上传、预览 Worker、BullMQ、Redis、Prisma、Socket 全链路 | 先讲业务后端工程，AI 当「加分业务模块」 |
| **主投** | 全栈工程师（Vue3 + Node） | 前后端 + 流式问答 + 网盘交互都有 | 「端到端交付过可演示产品」 |
| **主投** | AI 应用工程师 / 大模型应用开发 | RAG、Embedding、摘要、结构化抽取、ASR/TTS、VL、FC 助手 | 强调编排、异步索引、Provider 抽象，**别往算法靠** |
| **可投** | 知识库 / 智能文档 / RAG 业务研发 | 索引流水线、语义搜索、知识卡片、知识库 | 用「上传→分块→向量→检索→回答」讲清架构 |
| **可投（偏业务）** | B 端 / 工具类产品后端 | 文件、分享、VIP、查重、搜图都是典型 ToB/工具场景 | 业务复杂度 + AI 差异化 |

### 3.1 不建议当主投

| 岗位 | 原因 |
|------|------|
| 算法工程师 / NLP 研究员 | 无训练、无论文向证据，面试易被打穿 |
| 纯前端 | 项目叙事重心在 Nest + AI 流水线 |
| 纯运维 / SRE | Docker Compose 够用，缺大规模运维故事 |

## 4. 面试时已经能讲的

这些在「AI 应用」面试里区分度高，且是**真做过**：

1. **RAG 全链路**：抽取 → chunk → embedding → Top-K → 流式回答  
2. **异步索引**：BullMQ Worker，与预览转码同类工程模式  
3. **多模态业务**：扫描 PDF / OCR、VL 解题、以图搜图、ASR segments  
4. **工程分层**：ASR / TTS / Embedding 等 Provider 与业务域拆开；目录重构可讲可维护性  
5. **结构化输出**：`generateObject` / Schema 抽知识点  
6. **Function Calling 助手 MVP**：只读工具调用（Agent 入门故事）

## 5. AI 方面还缺什么（按优先级）

原则：**面试被问到的概率 × 补完收益**；只补高频，不追冷门。

### 5.1 向量库与检索进阶（最高优先）

**现状：** MySQL JSON + 应用层余弦相似度——MVP 够用。  
**风险：** 面试几乎必问「为什么不上向量库 / 怎么扩」。

**补到能讲清楚：**

- `pgvector` 或 `Qdrant` 其一，做一次迁移对比（延迟、召回、运维）  
- Hybrid：关键词 + 向量；简单 rerank（交叉编码器或 LLM rerank 二选一）  
- 分块策略取舍（固定长度 vs 章节感知）及失败案例  

### 5.2 Agent / Tool 编排（第二优先）

**现状：** 只读 FC MVP；JD 里「Agent」很热。

**补到能讲清楚：**

- 多工具：查文件 / 搜知识库 / 触发摘要，带权限边界  
- 失败重试、循环上限、幻觉时「拒答」  
- （可选）MCP 暴露网盘做一版最小 demo——话题新，但不要做成主项目压过核心业务  

### 5.3 RAG 评测与质量（第三优先）

拉开和「调过 API」候选人的差距：多数人讲不清「怎么知道 RAG 好不好」。

**补到能讲清楚：**

- 一小套黄金问答集（20～50 条）  
- 指标：命中率、引用是否落在 chunk 内、拒答率  
- 1～2 个真实翻车案例 + 如何改 prompt / 分块  

### 5.4 生产向：配额、成本、可观测（第四优先）

**现状：** PRD 中 F-18 限流 / VIP 配额押后。  
**风险：** 面试官常问「API 烧钱怎么办」。

**补到能讲清楚：**

- 按用户 / VIP 限流与日配额  
- token / 调用量日志  
- 超时、熔断、降级（关 RAG 只留划词）

## 6. 够用即可、别深挖

| 话题 | 建议 |
|------|------|
| LangChain / LlamaIndex | 能对比「为何自研流水线」即可，不必重写整站 |
| 微调 / 训练 | 明确不做，一句话挡回去 |
| K8s 大规模集群 | 有 Docker Compose 就够，别硬装运维 |
| 自研 ASR / VL 模型 | 继续云 API + Provider，讲清替换成本 |

## 7. 投递策略

1. **简历标题：** `NestJS 后端 / 全栈（AI 应用）`，不要写「算法」。  
2. **项目条建议 3 条可量化：** 异步索引队列；RAG 问答；ASR→segments→字幕/摘要；再补 1 个数字（文件类型数、测试套件数、端点数等）。  
3. **主投比例建议：** 后端 / 全栈约 60%，AI 应用约 30%，知识库类约 10%。  
4. **冷门岗：** 不投；岗位名以 BOSS / 拉勾常见 JD 为准。
