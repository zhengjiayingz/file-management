## 问答一：全库两两算余弦会不会算爆？

### 问题

> 现在想做近似查重的话，如果进行循环算相似度的话，会不会很耗性能？  
> 比如网盘里有 10000 个文件，那么 2 层循环要算 1 亿次余弦相似度？

### 详细回答

**如果真对全库做「每个文件和每个文件比余弦」，量级就是 \(O(n^2)\)，会很耗性能——而且不该这么做。**

更准确的数量级：

| 比法 | 大约次数（\(n=10000\)） |
|------|-------------------------|
| 有向两两（\(i\) 对 \(j\)，含顺序） | 约 \(n^2\) ≈ **1 亿** |
| 无向两两（只比一次 \(i<j\)） | 约 \(n(n-1)/2\) ≈ **五千万** |

若每次还是高维向量（成百上千维）的余弦，CPU 与接口耗时都会很难接受。

**这和「以图搜图」不是同一类问题：**

- 以图搜图：1 个查询向量 × \(N\) 张候选图 → **\(O(N)\)**（你们还用 `MAX_GALLERY` 截断）  
- 查重若两两比：每个都要和每个比 → **\(O(N^2)\)**

所以查重正确思路是：**先用便宜方法缩小范围，再在小范围内精比**，而不是全库嵌套循环算余弦。

你们方案里的分层（详见下一问）正是为了避开这个坑：

1. **content hash** 相同 → 直接成「完全重复」组（扫一遍分桶，约 \(O(n)\)）  
2. **图片 pHash** 先找「看起来很近」的候选  
3. **向量余弦**（若需要）只在候选对 / 小桶内算，而不是 1 万 × 1 万  

面试也可以讲：知道 \(n^2\) 不可行，所以用「指纹分层 + 分桶」，而不是暴力余弦。

## 问答二：pHash 等术语听不懂，前置知识是什么？方案在做什么？

### 问题

> 没有看懂「分桶 / pHash」那些解释。  
> 先把术语涉及到的前置知识讲清楚，再讲这个查重方案具体在做什么。

### 前置知识（从零）

#### 1. 什么是「指纹」？

指纹 = 给文件算出来的一串**短特征 / 身份摘要**，用来比较「像不像」，而不是每次把整个文件内容两两比对。

| 指纹类型 | 一句话 |
|----------|--------|
| **内容 hash（content hash）** | 文件字节一模一样 → 码一样 |
| **pHash** | 图看起来几乎一样（哪怕压缩、缩放一点）→ 码很接近 |
| **向量 embedding** | 语义/观感相似 → 在高维空间里距离近 |

项目里已有的 `FileFingerprint` / `fileHash` 等，就是在落这类特征（查重会扩展 pHash、文本指纹等）。

#### 2. 内容 hash（content hash）是什么？

对文件**全部字节**做哈希（如 MD5、SHA256），得到固定长度字符串。

- 改 **1 个字节** → hash **完全变**（雪崩）  
- 两个文件 hash 相同 → 基本可认定 **完全一样**（精确重复）  
- 工程上：扫一遍，用 `Map<hash, 文件列表>` 分组即可，**不需要两两循环**

你们上传链路里已有按内容算的 `fileHash`（见存储 / 秒传相关逻辑），查重精确层可以直接复用或写入指纹表。

#### 3. pHash 是什么？

**pHash = perceptual hash（感知哈希）**，主要给**图片**用。

它不关心「文件字节是否完全一样」，而关心「人眼看起来像不像」。

举例：

- 同一张图：原图 jpg / 压缩后 / 缩略图 → **字节不同**，content hash **不同**  
- 但画面差不多 → pHash **相同或非常接近**

怎么比「接近」：

- pHash 通常是一串 **0/1 比特**（如 64 位）  
- 两张图比 **汉明距离**（有多少位不一样）  
  - 距离 0：几乎同一张  
  - 距离很小（如 ≤ 3）：很像  
  - 距离很大：不像  

| | content hash | pHash |
|--|--------------|-------|
| 依据 | 原始字节 | 画面感知 |
| 差一点点内容 | 完全不同 | 往往仍接近 |
| 典型用途 | 精确重复 | 图片近似重复 |

#### 4. 向量 / embedding / 余弦（项目里已用过）

以图搜图（F-38）已做过：

- 图片 → 一长串数字（向量）  
- 两个向量越「同向」→ 余弦相似度越高 → 越像  

查重里也可以用向量找「pHash 还不够近、但观感仍像」的图，但**更贵**，所以一般不当第一层。

#### 5. 「分桶」是什么意思？

像快递按邮编分区：

1. 先按某个粗特征分成很多小堆（桶）  
2. **只在同一堆里**仔细比  

这样总比较次数从「全体 × 全体」变成「各小堆内部」。

- content hash 相同 → 天然一个桶  
- pHash 也可用「前缀相同」或「距离够近」进入同一候选堆  

### 查重方案具体在做什么？

**产品目标：** 找出「完全重复 / 近似重复」的文件组，在独立页列出，用户多选后手动删除；**系统不自动删**。

流水线可以记成：

```text
网盘文件
  │
  ├─① 算指纹（content hash / pHash / 可选文本或图像向量）存起来
  │     （新上传可自动；历史可「扫描全部」补算）
  │
  └─② 用户点查重 / 扫描时按指纹分组
        │
        ├─ 完全重复组：content hash 相同
        ├─ 图片近似组：pHash 很近（必要时再对候选算图像向量余弦）
        └─ 文本近似组：文本指纹很像（SimHash 或 embedding 余弦）
              │
              └─③ 页面展示「重复组」→ 用户勾选 → 走现有删除/回收站
                    （建议至少保留组内 1 个）
```

**生活比喻：**

| 概念 | 比喻 |
|------|------|
| content hash | 身份证号：完全同一人 |
| pHash | 长相：双胞胎 / 同人不同发型 |
| embedding 余弦 | 气质/题材：都是「海边小狗」类照片 |
| 分桶后再比 | 先按班级分组，再在班里找像的，而不是全校两两比 |

## 问答三：什么是文本指纹？「abcd」和「abce」怎么判相似？

### 问题

> 什么是文本指纹？具体说明一下。  
> 假设两个 txt，内容分别是 `abcd` 和 `abce`（微小区别），指纹大概会是怎样的？  
> 怎么通过文本指纹判断内容非常相似？

### 详细回答

**「文本指纹」不是某一种固定算法**，而是「把整段文字压成一段短特征，用来比像不像」的统称。常见两类：

1. **哈希类**（如 SimHash）→ 比汉明距离  
2. **向量类**（文本 embedding）→ 比余弦相似度（更接近你们 AI 网盘的做法）

精确重复仍用 **content hash**；近似才用文本指纹。

#### 先排除：普通 content hash 对近似没用

对全文做 MD5/SHA：

| 内容 | hash（示意） |
|------|----------------|
| `abcd` | `A1B2...` |
| `abce` | `9F3C...`（完全不同） |

差一个字母 → hash 面目全非 → **无法**说「很相似」。  
所以：只改名的复制件靠 content hash；「改了一个字」的近重要靠文本指纹。

#### 做法 A：SimHash 类（经典比特指纹）

思路（极度简化）：

1. 把文本拆成小片段（字 / 词 / n-gram），如 `abcd` → `ab`, `bc`, `cd`  
2. 每个片段变成比特，加权合成固定长度比特串（如 64 位）  
3. 两个指纹比 **汉明距离**（有多少位不同）  
   - 距离很小 → 很像  
   - 距离很大 → 不像  

对你这个例子（**示意，不是真实算出的值**）：

| 文本 | SimHash（示意 8 位） |
|------|----------------------|
| `abcd` | `11010010` |
| `abce` | `11010011` |

汉明距离 = 1 → **非常相似**。  
若 `abcd` vs `zzzz`，可能变成差很多位 → 不相似。

**判断规则示例：** 64 位指纹时，汉明距离 ≤ 某阈值（如 3）→ 进同一近似组。

#### 做法 B：向量指纹（embedding）

思路：

1. 用文本 embedding 把全文（或截断 / 摘要）编成向量  
2. 算余弦相似度  
3. 相似度 ≥ 阈值 → 近似重复  

示意：

| 文本 | 向量（只写前几维示意） |
|------|-------------------------|
| `abcd` | `[0.12, -0.05, 0.88, 0.31, ...]` |
| `abce` | `[0.11, -0.04, 0.87, 0.30, ...]` |

几乎同向 → 余弦可能 **0.99+** → 判相似。  
与完全无关长文 → 余弦可能只有 **0.2～0.4** → 不相似。

**判断规则示例：** `cosine ≥ 0.92`（阈值可配，如 `DEDUP_TEXT_COSINE_MIN`）→ 同一近似组。

注意：短到只有 4 个字母时，模型行为可能不稳定；产品上常会加「太短不参与近重」之类规则。

#### 判断流程（一句话）

```text
1. 每个文本文件 → 算出文本指纹（比特串或向量）存库
2. 查重时：
   - 精确：content hash 相同 → 一组
   - 近似：比指纹
       · SimHash：汉明距离小
       · Embedding：余弦大
3. 超过阈值 → 「重复组」展示给用户
```

仍不建议 1 万文件两两全比：可先按长度分桶、MinHash / 前缀分桶，或 ANN，再在小范围精比。

#### 和方案文档的对应

方案中的「文本指纹」主要指：不必走完整 RAG `DocumentChunk` 表，也可为查重单独存一份短文本 / 全文特征（embedding 或同类），用余弦做近似；精确层仍然靠 content hash。扫描 PDF 若只为查重，可「只 OCR + 指纹」而不建完整 RAG。

## 问答四：只改文件名，content hash 会相同吗？

### 问题

> 近似查重也要做。另一个问题：假设一个文件是从另一个文件复制出来的，只是改了名字，这两个文件的 content hash 会相同吗？

### 详细回答

**会相同。**

content hash（你们库里常见字段是 `fileHash`）按**文件内容字节**计算，一般**不包含**文件名、路径、上传时间。

| 操作 | content hash |
|------|----------------|
| 复制文件，只改名（`a.txt` → `b.txt`） | **相同** |
| 改文件里任意一个字 | **不同** |
| 同一内容，扩展名改了但字节完全一样 | **相同** |
| 重新保存导致换行符 / 编码变了 | **可能不同** |

因此：

- **精确查重**正是靠 content hash：同 hash → 「完全重复」组  
- 「复制出来只改名」是最典型的完全重复场景  
- **近似查重**覆盖的是 hash 已经不同、但内容/画面仍很像的情况（改了一点字、压缩图等）

产品上两层都要做：

1. **精确**：content hash 相同 → 完全重复  
2. **近似**：hash 不同，但 pHash / 文本指纹很像 → 近似重复

## 和本项目查重路线的对应

| 能力 | PRD / 方案 | 技术侧重点 |
|------|------------|------------|
| 完全重复 | F-35 精确层 | content hash / `fileHash` 分桶，\(O(n)\) |
| 图片近似 | F-35 | pHash 汉明距离；可选图像向量只比候选 |
| 文本近似 | F-35 | 文本指纹（SimHash 或 embedding 余弦） |
| 指纹写入时机 | 方案 §1.3 | 新上传默认可参与；「扫描全部」补历史 |
| UX | 方案 §1.4 | 独立页、分组列出、多选手动删、不自动删 |
| 不做 | 方案 §1.7 | 自动删、全库暴力余弦、本期不上独立向量库也行（千级内存聚类可 MVP） |

相关实现落点（后续开发时）：

- 数据：扩展 `FileFingerprint`（或等价表）增加 `contentHash` / `pHash` / `textFingerprint` 等  
- API：如 `POST /api/files/duplicates/scan`，返回 `groups[]`  
- 前端：侧栏「重复文件」独立页

## 术语与决策速查

| 术语 | 含义 | 查重里怎么用 |
|------|------|----------------|
| content hash | 内容字节摘要 | 精确重复；只改名仍相同 |
| pHash | 图片感知哈希 | 图片近似；比汉明距离 |
| 文本指纹 | 文本短特征（SimHash 或 embedding） | 文本近似 |
| 余弦相似度 | 两向量夹角度量 | 向量近重；勿全库两两 |
| 分桶 | 先粗分堆再精比 | 把 \(O(n^2)\) 压到可接受 |
| 汉明距离 | 两比特串有多少位不同 | pHash / SimHash |

**决策口诀：**

1. 先 hash 抓「复制只改名」  
2. 再 pHash / 文本指纹抓「差一点」  
3. 向量只打补丁，不打全库二层循环  
4. 结果分组展示，删不删用户说了算
