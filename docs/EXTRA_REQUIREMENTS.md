# 额外需求（可行性评估用）

本文档汇总网盘类产品中**常见、不偏门**的增量能力，供与现有 [REQUIREMENTS.md](./REQUIREMENTS.md) 对照后做**优先级与可行性**评估。  
每条均包含：**需求详细描述**（通俗说明「到底要做什么」）、**价值**、**建议范围**、**实现时关注点**，便于产品与技术评审、排期。

## 1. 两步验证（2FA / TOTP）

> **与本仓库实现的关系（2026-04-22）**：**全站/全角色** 的 2FA 仍属本文档级的产品扩展评估。**管理员账户** 已支持 **标准 TOTP**（可选、设置中启闭；`users.totp_enabled` 等；登录 `MFA_REQUIRED` → `mfa/verify`），以 [REQUIREMENTS.md](./REQUIREMENTS.md) **§5(8)** 与 [BUSINESS_FLOWS.md](./BUSINESS_FLOWS.md) **§1.2.1** 为准。下表「复杂度」等仍指若扩展到**所有用户**时的工作量，不等同于当前仅管理员一层的实现成本。

### 需求详细描述
除了输入**账号和密码**之外，再增加一道验证。用户用手机里的**身份验证器 App**（如 Google Authenticator、微软 Authenticator 等）扫一次二维码完成绑定；之后每次登录，在密码正确后还要再输入 App 里**每 30 秒变一次**的 6 位数字，系统核对正确才允许进入网盘。这样别人即使偷了你的密码，没有你的手机也登不上。可选：给用户一批**备用码**，手机丢了也能紧急登录一次并换绑。

### 价值
- 密码泄露或撞库时，第二因素可阻断大部分未授权登录。
- 企业、高校及对安全敏感的个人用户中，已成为**默认预期**；可作为 VIP 或「安全中心」能力宣传。

### 建议范围
- 用户可选开启 TOTP（兼容常见验证器 App）。
- 登录流程：账号密码通过后，若已开启 2FA，则校验动态码。
- 备用恢复码（防丢手机）、关闭 2FA 时的二次确认（密码 + 邮箱/短信择一，视现有能力）。

### 实现关注点
- 与现有双 Token、会话、`session_version` 失效策略对齐；防止「只验密码就发 Token」的绕过。
- 管理员重置用户密码时，是否要求重新绑定 2FA（产品策略）。

## 2. 共享文件夹 / 协作空间

### 需求详细描述
现在常见做法是「生成一个文件的分享链接」给别人。本需求是指：把整个**文件夹**共享给指定用户（或一个小团队），对方登录自己的账号后，在网盘里能直接看到**你授权给他的那个目录**里的内容，可以按权限**只看、或能上传、或能管理**。适合长期一起改资料、交作业、放公共素材，而不用每个新文件都重新发链接。谁邀请、谁退出、权限变更，都在文件夹维度统一管理。

### 价值
- 多人**持续**协作时，文件夹级权限比「每个文件一条分享链接」更易管理。
- 权限面更清晰：可统一撤销、统一可见范围，降低误分享风险。

### 建议范围
- 文件夹维度：邀请用户/角色（只读 / 上传 / 管理）或生成「成员链接」（择一或组合）。
- 与现有「好友」「消息」可联动通知，非必须。

### 实现关注点
- 与现有 `UserFile`、文件夹树、存储配额统计的一致性（谁占用配额：所有者 vs 协作者）。
- 分享链接与「协作成员」两种模型是否并存，需统一语义避免重复入口。

## 3. 全文检索（多类型文档）

### 需求详细描述
搜索框不只搜**文件名**，还能搜**文件里面的文字**。例如记得某份笔记里写过「预算」两个字，但忘了文件叫什么，输入「预算」能把含有该词的 txt、Markdown、（可选）PDF 等找出来。后台需要在文件上传或更新后，把可读文本**抽出来建索引**；用户搜索时返回匹配的文件列表，必要时带一小段上下文摘要。

### 价值
- 用户记不住文件名时，可按**正文关键词**定位文件。
- 与标签、目录并列，提升「长期资料沉淀」的留存。

### 建议范围
- **一期**：纯文本、Markdown；可选 PDF（文本层）。
- 异步索引：上传/更新后入队，避免阻塞上传接口。
- 检索 API + 前端结果列表（高亮片段可选）。

### 实现关注点
- 存储与算力：自建简易索引 vs 引入 Elasticsearch 等；需评估实例成本。
- 隐私：是否仅索引「用户本人文件」、加密文件是否排除索引（产品声明）。

## 4. 分享链接加固（密码、次数/流量、仅预览）

### 需求详细描述
在现有「分享链接 + 可选提取码 + 过期时间」基础上，让分享更**可控**：例如打开链接时先要再输一道**独立密码**；或这个链接**最多被访问 / 下载 N 次**就用完；或**总共最多下载多少流量**就停用；或只允许在网页里**在线看**（预览），不提供「另存为」的下载入口（需技术上加固，避免用接口地址绕过）。适合把资料发给客户、学生时，减少链接被到处转发后的失控。

### 价值
- 外发资料（课件、报价、合同草稿）时可**限次、限流、防直链滥用**。
- 「仅预览不可下载」满足部分版权/防扩散场景（需配合预览能力）。

### 建议范围
- 链接级：可选密码、最大访问次数、最大下载字节数（或单日上限）。
- 与现有 `max_visitors`、过期时间、提取码模型**合并设计**，避免字段爆炸。

### 实现关注点
- 仅预览需可靠区分「预览接口」与「下载接口」，防止 URL 拼接绕过。
- 统计口径：按 IP / 登录用户 / Cookie 设备指纹（与现有一致即可）。

## 5. 存储配额预警与通知

### 需求详细描述
网盘会给每个用户一个**容量上限**（例如 2GB）。本需求是：当已用空间接近上限时（例如用到 80%、95%），系统自动给用户发**提醒**（站内消息为主，邮件可选），必要时告知「再传多少会满」。避免用户传大文件传到一半才发现空间不够，或突然不能同步却不知道为什么。若做管理员视图，还可看全站存储走势，便于扩容。

### 价值
- 在 80% / 95% 等阈值提醒，减少**突然无法上传**的挫败感。
- 管理员侧可看全局存储趋势（可选），便于运维与扩容决策。

### 建议范围
- 触发：配额计算后异步或定时检查，写站内消息；邮件为可选扩展。
- 用户设置：开关提醒、阈值（可选，简化可先固定阈值）。

### 实现关注点
- 与 `storage_used` 更新频率一致，避免重复骚扰（冷却时间或每日最多一条）。

## 6. Office 类在线预览 / 可选编辑

### 需求详细描述
用户在网页里点开 Word、Excel、PPT、PDF 等，**不先下载到电脑**，直接在浏览器里**像在线文档那样打开看**（一期目标）。若能力允许，二期可支持**简单编辑**并保存回网盘（会涉及版本记录）。通常对接成熟的在线文档引擎或云服务，减少自研成本。和「分享链接只允许预览」搭配时，对方点开链接即可阅读，降低文件被随意下载扩散的风险。

### 价值
- 减少「下载 → 本地编辑 → 再上传」路径，提升日活与 VIP 付费理由。
- 与「仅预览分享」组合时价值更高。

### 建议范围
- **一期**：对接成熟方案（如 OnlyOffice / Collabora / 第三方 SaaS）仅预览。
- **二期**：可编辑 + 回写版本（与现有文件版本管理衔接）。

### 实现关注点
- 文档暂存与密钥：是否经自有服务器中转；隐私条款与用户协议。
- 授权：分享链接打开预览时的 Token 与短时效。

## 7. Webhook / 文件夹事件

### 需求详细描述
用户或管理员在网盘里**指定一个文件夹**，并填写一个**自己的服务器地址（URL）**。当这个文件夹里发生约定好的事情时（例如**新文件上传、文件被删、文件被更新**），网盘服务器自动向该 URL **发一条 HTTP 请求**（通常带 JSON 说明是谁、哪个文件、什么操作），对方系统收到后可以做备份、杀毒、审批、统计等自动化。类似「网盘主动通知外部系统」，面向会用接口的开发者或小团队。

### 价值
- 指定目录「文件创建/更新/删除」时向业务 URL POST JSON，便于备份、审批、转码等**自动化**。
- 面向小团队与开发者，提高平台「可集成」印象。

### 建议范围
- 用户或管理员配置：URL、密钥（HMAC 签名校验）、订阅事件类型、重试策略。
- 与操作日志区分：Webhook 为出站，需限流与失败队列（简版可先同步 HTTP + 日志）。

### 实现关注点
- 安全：防 SSRF（限制内网 URL）、签名与时间戳防重放。
- 可靠性：异步队列（Redis / DB 表）优于纯同步请求。

## 8. 数据可携带 / 账号数据导出

### 需求详细描述
用户想**带走自己在网盘里的数据**时，提供官方能力：例如一键生成**文件列表**（表格），或申请**打包下载**（后台慢慢打成 ZIP，好了给限时下载链接），把「我的文件」整体拷走。进阶可带上目录结构、标签、分享记录等元数据。用户感觉是「数据是我的，平台不绑架」；对外也可说是隐私与合规方面的基础能力。

### 价值
- 满足隐私合规叙事，降低「数据锁死」顾虑，提升信任。
- 企业采购时常问的「能否拿走数据」。

### 建议范围
- **一期**：导出文件清单（CSV/JSON）+ 批量打包下载任务（异步生成 ZIP 链接）。
- **二期**：包含分享记录、标签、目录结构元数据。

### 实现关注点
- 大账号分包与过期链接；审计日志记录导出行为。
- GDPR/个保法口径需法务确认，文档层面前置说明即可。

## 9. 回收站与分享联动策略

### 需求详细描述
当用户把某个文件**删进回收站**或**彻底删除**时，如果该文件之前被**分享过**，分享链接不能再让别人正常下载到有效内容，应**自动失效**或在打开时明确提示「文件已删除」。从回收站**恢复**文件后，是自动恢复原来的分享，还是要求用户重新开分享，需要产品定规则并在界面上写清楚。避免出现「链接还能打开但文件没了」或引发纠纷的**幽灵链接**。

### 价值
- 文件进回收站或彻底删除时，相关分享链接**自动失效或显式提示**，避免访问死链或法律纠纷。
- 恢复文件时是否恢复分享状态，需产品规则清晰。

### 建议范围
- 定义状态机：删除 → 分享 `disabled`；恢复 → 可选恢复分享或需用户重新开启。
- 前端分享管理列表展示「因文件删除已失效」原因。

### 实现关注点
- 与秒传、多引用同一物理文件的逻辑一致：删除引用不等于删物理文件时，分享应指向用户视图而非物理文件。

## 10. 大文件 / 弱网体验强化

### 需求详细描述
在上传**很大**的文件，或**网络不稳定**（地铁、校园网、热点）时，界面要清楚显示**进度、已传多少、失败原因**；断线或关页面后，应能**从断点接着传**，而不是从头再来。可配合**自动重试**、并发分片数量随网络调整等。目标是：普通用户不觉得「大文件一传就废」，移动端和弱网场景也能用。

### 价值
- 校园网、地铁等弱网与移动端场景下，**可恢复、可预期**的上传体验直接决定口碑。
- 与现有分片上传能力形成闭环（重试、断点、限速提示）。

### 建议范围
- 前端：统一进度、失败原因、一键续传；后台任务可查询分片状态。
- 可选：客户端并发分片数自适应网络；服务端限流与友好错误码。

### 实现关注点
- 与浏览器休眠、页面刷新后的续传一致性；合并任务与数据库事务边界。
