# WebSocket / Socket 常见面试题与参考答案（详细版）

> 本版答案尽量**写清谁对谁做了什么**（主谓宾完整），并串成**可口述的流程**。可结合本仓库：Socket.IO、路径 `/socket.io`、握手 `auth.token`、先写 `messages` 再 `emitToUser` 等。

---

## 一、基础概念

### 1. WebSocket 和 HTTP 有什么主要区别？

**答（建议口述结构：先各用一句话，再对比）：**

- **HTTP** 的默认使用方式是：**由客户端（常见是浏览器或 App）向服务器发起一个请求，服务器返回一个响应，然后一次交互在语义上可以结束**；即使用户开了「保持连接」（HTTP/1.1 的 `keep-alive`），**在应用层协议语义上，多数接口仍然是「请求一次、响应一次」的样式**，而且**服务器**通常**不能在没有请求的情况下，主动向浏览器用「同一个 HTTP 语义」去推送**（长轮询、SSE 等是变通手段，不是标准 HTTP 最直观的那条路径）。

- **WebSocket** 是：**在一条 TCP 连接上，在先用 HTTP 做了一次「特殊握手」之后，把这条连接升级成一种「双方谁都可以先说话」的协议**；建连之后，**服务器可以在任意时刻，在没有新的 HTTP 请求的情况下，给浏览器发数据，浏览器也可以随时给服务器发数据**。

- **因此**：聊天、状态推送、协同编辑等「服务器要主动推」、且客户端也要频繁上行的场景，**用 WebSocket 更自然**；而鉴权、文件上传、普通 REST 业务，**继续用 HTTP 很常见**；实际项目里**两种协议并存**是常态。

---

### 2. 连接是怎么建立的？（和 HTTP 的关系）

**答（分「谁发起什么」一步步说，面试官最爱听这个）：**

1. **谁发起第一次请求？**  
   **你的浏览器里运行的 WebSocket 客户端**（对原生 API 就是 `new WebSocket(url)`；用 Socket.IO 时则是 **`socket.io-client` 的 `io(url)` 函数**在内部帮你发）**向你的服务器发起一次普通的 HTTP 请求**。

2. **这次普通 HTTP 请求里有什么特别？**  
   这个请求在 HTTP 头里会带上**升级意图**，典型是 `Connection: Upgrade` 和 `Upgrade: websocket`（**这就是常说的「HTTP Upgrade 请求」**），意思是：**我不想一直当普通 HTTP 短对话，我想和你在同一条 TCP 上换成 WebSocket 协议**（若是 Socket.IO 第一次也可能是长轮询的 HTTP 请求，**视客户端与服务端协商结果而定**）。

3. **服务器在做什么？**  
   **你的 Web 服务器**（如 Node 里挂在 `http.Server` 上的逻辑，或 Nginx 再反代到 Node）**如果同意升级**，会返回 **状态码 101**（`Switching Protocols`），表示**握手成功**；从这一刻起，**这条连接上的数据格式**就不再按「一请求一响应」的 HTTP 文档形态为主，**而是切到 WebSocket 的帧（frame）格式**，由**客户端与服务器**双方**随时互相发送帧**。

4. **握手之后谁还可以先发？**  
   **任一方**都可以先发 WebSocket 帧，所以叫**全双工**；这与「**必须由浏览器先点一次才会有一次 HTTP 请求**」的直觉不完全一样，因为**同一条长连接**已经**保持**在那里了。

5. **Socket.IO 和「纯 WebSocket」差在哪（面试可补一句）？**  
   **Socket.IO 的客户端**往往**第一次不一定直接 Upgrade**，可能先用 **HTTP 长轮询** 把隧道拉起来，**再**在同一个会话里**升级到 WebSocket**；**传输顺序**上你可以记：**优先 WebSocket，不行再降回轮询**（你项目里 `transports: ['websocket', 'polling']` 就是这个意思）。

6. **若 Upgrade 失败会怎样？**  
   **连接不会变成 WebSocket**；**Socket.IO** 会**退回到**「用 HTTP 长轮询模拟实时」之类的路径，**所以业务代码仍然能通**，只是**延迟和资源占用**通常**不如**真正 WebSocket 好。

---

### 3. 为什么需要「心跳」？ping / pong 是干什么的？

**答：**

- **长连接**建立之后，在公网、家庭路由器、公司防火墙、**负载均衡**、云厂商 NAT 上，**长时间「看起来没有数据」**的连接，中间设备**可能把这条连接悄悄扔掉**，**但浏览器和 Node 并不立刻知道**（这叫**半开连接**或**僵尸连接**的某种形态）。

- **心跳**就是：**在固定时间间隔里，让客户端或协议栈主动发一个「很小的包」**；对端**回了**，双方就知道**对端还活着、中间设备还没把路掐断**。WebSocket 协议里也有 **ping 帧 / pong 帧** 的概念，**由库或浏览器帮你处理一部分**；业务里也可以**自己发应用层「心跳消息」**。

- **若心跳超时**，**客户端的 Socket 库**通常会**发现连接不可靠**，**触发重连**或**交给业务处理**，避免用户界面以为**还连着**，其实**发不出去、也收不到**。

---

### 4. 断线之后一般怎么处理？

**答：**

- **Socket.IO 客户端**通常**自动尝试重连**（**谁重连**？**你页面里的 `io()` 返回的 socket 对象背后的逻辑**在重连；**你业务代码**常见是**监听** `connect` / `disconnect` / `reconnect` 来更新状态）。

- **重连策略**上，**不能无限疯狂连**，一般会**加间隔（指数退避）和最大次数**，否则**会打满服务器、害自己也害别人**。

- **重连成功之后，身份还能用吗？**  
  若** JWT 没过期**，**旧 token 往往还能**继续握手；**若已过期**，**你业务里应该**像 HTTP 一样，**用 Refresh 换新 Access**，**再** `connect`（你项目可对照「存不存在登录后没换 socket token」的改进点，面试可说「知道要换 token 再重连」）。

- **重要一条**：**聊天消息是否已送达、历史记录是什么**，**以服务器数据库里存的为准**；**重连后**用 **HTTP 拉未读/拉历史** **补齐**，**不要**假设「我 socket 上一条都拿到了」。

---

### 5. 什么是全双工、半双工？

**答：**

- **半双工**指**同一条通路上，同一时刻**通常**只允许一个方向的「有效说话」**（对讲机是经典比喻）。

- **全双工**指**同一条通路上，同一时刻，双方**都可以**发数据**，**不互相等对方发完**（电话是常见比喻；**单条 WebSocket 连接在协议能力上是全双工**）。

- 你可以这样背：**单条 WebSocket 连接 = 全双工**；**单条传统 HTTP/1.1 请求-响应 = 在单次请求里先客户端后服务器，是「交互意义」上的串行，不是同一条可持久双向比特流**（这里别和物理层全双工混成一道题，面试说清楚「应用里我们把 WebSocket 当双向通道」即可）。

## 二、与轮询、SSE、长轮询的对比

### 6. 什么时候用 WebSocket，什么时候用「短轮询」就够？

**答：**

- **短轮测**是：**浏览器**定时用 **HTTP** 问服务器「有没有新东西」；**实现简单**，**但**有「新东西」的间隔很密时，**空轮询**会浪费**请求数**、**电**、**服务器连接**。

- **你什么时候能只用短轮询**？**数据更新不频繁、晚几秒知道也没关系、或者团队不想维护长连接**，短轮询就够用（例如**管理后台的列表每 30 秒刷新**）。

- **你什么时候会转向 WebSocket**？**新消息要「尽快」到、聊天双方**都要**高频发**、**游戏状态**、**协同光标** 等；此时**用长连接**比**用大量 HTTP 轮询**更省总请求、更自然。

---

### 7. SSE（Server-Sent Events）和 WebSocket 怎么选？

**答：**

- **服务器** 用 **HTTP 长连** 或 **分块** 把 **只从服务器到浏览器的** 流式事件推给客户端；**浏览器** 用 `EventSource` 这个 API 接；**方向** 只有 **「服务器向浏览器推」** 这一种主力方向（**所以叫 SSE，单向**）。**重连、事件 id** 在规范里**对浏览器有较好支持**。

- **WebSocket** 是 **双向**；**你既要收又要经常发**（**聊天、游戏指令**）时，**用 WebSocket**。

- 一句结论：**只读类实时、只要服务器推**且**不介意单向**，**SSE 往往更简单**；**双向**选 **WebSocket**。

---

### 8. 能不能用 WebSocket 完全替代 HTTP？

**答：**

- **不能也不应该**在一般业务里**完全替代**。**文件上传/下载、开放 API 给第三方、搜索引擎索引、Cache、幂等的 GET、REST 的语义**，**用 HTTP/HTTPS** 是**业界最成熟、工具链最完整**的。

- **你们实际工程里常见**的是：**用户登录、发消息、拉历史**走 **HTTP + 数据库**；**「对方在线时立刻知道」**走 **WebSocket 推送**；**两端分工清晰**。

---

## 三、工程实现（与 Node / Socket.IO 常考）

### 9. 同一服务里 HTTP 和 WebSocket 能共用端口吗？怎么做的？

**答：**

- **能共用**。在 **Node** 里，**你先** 用 **`http.createServer( expressApp )` 或 `http.createServer( 回调 )` 造出一个** **HTTP 服务器对象**；**同一份** **TLS 的话是 `https.createServer`**，思路一样。

- **然后** **你把（例如）Socket.IO 的 `Server` 挂在这个已存在的 `httpServer` 上**；**这样** 外部客户端连 **同一个** **端口** 时，**既可以是普通 HTTP 请求**（`GET /api/...`），**也可以是** 走 **同端口** 的 **WebSocket 升级**（**由客户端 Upgrade 和服务器 101 决定**）。

- **本仓库** 里 **`app.ts` 写的就是**：**先** `createServer(app)` 得到 `httpServer`，**再** `initSocket(httpServer)`，**最后** `httpServer.listen(3000)`，**所以** **REST** 和 **Socket.IO** **共用 3000**。

---

### 10. WebSocket 连接上如何做鉴权？

**答：**

- **握手还没完全变成 WebSocket 时**，这次请求**还是 HTTP**，所以**你可以在这一阶段** 把 **JWT** 放在 **Query 字符串**、**Header**、**或 Socket.IO 的 `auth` 对象** 里，**让服务器上的中间件**（**例如** Socket.IO 的 `io.use`）**先验人**、**不通过**就 **`next(错误)`** **拒绝**这条连接，**这样** 就不会有一个「匿名 socket」混进来。

- **验什么**？和 HTTP 一样，常见是 **JWT 是否合法**、**用户是否被禁用**、**token 里的会话版本** 和 **服务器用户表** 里 **是否一致**（**你们**就是 **`sessionVersion` 与 `sv` 一致**），以及 **是否必须改密** 等**业务条件**。

- **连接建立后**，**发送聊天 payload 时**，**发消息人的 id** 必须**由服务端从已认证的会话**里取，**不能** 信客户端在 JSON 里写「我是 5 号用户」就当成 5 号（见第 21 题）。

- **本仓库** 的 **`realtime/socket.ts`**：**`io.use`** 里**读取** `socket.handshake.auth.token`，**验证** 后**把** `user.id` **放进** `socket.data.userId`，**后续** 推消息时**知道** 这条线是谁。

---

### 11. 什么是 Socket 的 room（房间）？用来解决什么问题？

**答：**

- **谁创建「房间名」**？**不是** 前端在浏览器里 `new` 一个对象；**是** 服务器在内存里，**对「某一类业务分组」** 起了一个**字符串名字**（**例如** `user:10086`），**让某些 socket 连接 `join` 进这个名字下面**。

- **谁 `join`？** **服务器代码** 在 `connection` 里**对** **当前** **这条** `socket` 调用 `socket.join('user:10086')`；**含义**是：「**这条** TCP 长连接，代表 10086 这个用户，**他接收推送时，请归在这个名为 `user:10086` 的组里**」。

- **发的时候谁收？** **服务器** 调用 **`io.to('user:10086').emit('某事件', 数据)`**，**则** 所有**已经 `join` 了 `user:10086` 的 socket**（**例如** 同一个人开了两个标签页、两条连接**都 join 了同一个名字**）**都会收到**；**这** 就解决了**「我按用户 id 广播，而不是按某一条物理连接去猜」** 的问题。

- **本仓库** 的 **`connection`** 里**写** `socket.join('user:' + uid)`，**发私信** 时 `emitToUser(对方id)` 往 **`user:对方id`** 推 **`message:new`**。

---

### 12. 多机部署时，A 机上的用户发消息，怎么推给在 B 机上的用户？

**答：**

- **单机** 上 **Socket.IO 的内存** 里，**能 `emit` 的** 只有 **连在本机** 的 socket；**在 B 机上的** 那一条 WebSocket 连接，**A 机进程** **根本见不着**。

- **所以** 必须有一种 **「跨机器」 的消息通路**：**常见** 是 **Redis Pub/Sub** 配 **Socket.IO 的 Redis Adapter**——**A 机** 说「**请** 给**用户 100** 发 `message:new`」，**这条指令** 被 **发布到 Redis**；**B 机** **订阅了** 后**在本机** 上执行 `io.to('user:100').emit(...)`，**B 上** 的 **100 号用户的连接** 才能收到。

- **另一种** 是**业务层不依赖本机 `io` 的内存**：**发消息** 进 **消息队列**；**一个专门的推送服务** 只负责 **消费队列 + 发 WebSocket**，**这** 里又涉及**服务拆分**。

- **面试** 时你要讲清：**多实例下「内存里的 io」不共享，必须 adapter 或集中推送服务。**

---

### 13. 什么是命名空间（namespace）？和 room 有何不同？

**答：**

- **namespace** 是 **URL 上更靠前的那一段路径差异**（**例如** `/admin` 与 `/`），在 Socket.IO 里**代表「一整套独立的 socket 空间」**；**不同 namespace** 的**连接与事件是隔离的**。

- **room** 是 **同一个** namespace 里面**再** 做的**更细** 分组，**一个用户、一个群** 都**可以** 是一个 room 名。

- **你一般 IM 项目** 可以 **一个默认 namespace 就够**；**只有** 当你要 **「同域名下两套完全不相干的长连业务、事件名还可能冲突」** 时，**再** 拆 **namespace**。

## 四、数据一致性与消息语义

### 14. 应该是「先推再存」还是「先存再推」？

**答：**

- **先存（数据库/持久化层）再推（WebSocket）** 的**好处**是：数据库**永远是** **权威真相**；**推丢了**，用户**用 HTTP 拉历史** 还能**对齐**；**先推了再存，若** **数据库写失败**，**对方手机上已经显示了一条「幽灵消息」**，**很难收拾**。

- **本仓库** 是 **先** `prisma.message.create` **再** `loadMessageForEmit` **再** `emitToUser`。

---

### 15. 消息的「至少一次、最多一次、恰好一次」在 WebSocket 里怎么理解？

**答：**

- **TCP/WebSocket 并不自动给你** **业务上** 的**恰好一次送达保证**；**你断线、重发、** **客户端** **重复点发送**，都可能导致**同一条业务消息** 在**感知上** 出现多次。

- **产品上要「恰好一次」或「不重复显示」** 时，**通常** 给**每条业务消息** 一个 **可去重的 id**（**客户端生成** `clientMessageId` 或 **服务端** **雪花/自增**），**服务端** **按 id 幂等** **写库**；**重连** 用 **时间戳/序号** 做 **增量拉**。

- **你面试** 说清：**传输层不保证，业务层用 id 与幂等去补**。

---

### 16. 对方离线时，消息放哪里？在线后再怎么收到？

**答：**

- **发消息** 的 **POST 请求** 在 **服务器** 上**写进数据库的 `messages` 表**；**这** 与对**方** 是否**连着 WebSocket** **无关**。

- **对** 若 **不在线**，**`emitToUser` 在内存里** **找不到** 那人的 socket，**推不到**，**但** **库里已有记录**；**对** 方 **后来** 打开 App，**用 HTTP** **拉** `/messages/:好友id` 的**历史** **即可看到**；**未读** 用 **表字段或汇总接口** 做。

- **你** 不要 **把「socket 没连上」理解成「消息没发出去」**；**只要 HTTP 发消息成功**，**以库为准**。

---

## 五、性能、扩展与排障

### 17. 长连接会消耗什么资源？单机能撑多少「连接数」？

**答：**

- **每一条** 长连接在 **OS** 上占 **一个** **socket** 描述符、在 **Node** 里占 **收发包缓冲区、JS 里对应 socket 对象**；**万级连接** 时**内存、文件描述符上限** 经常先成为**瓶颈**。

- **单机能连多少** **没有** 固定真理数字，**和** 机器 **CPU/内存/内核参数、心跳频率、是否每消息广播给所有人** 都有关；**面试** 上**说「要压测、要看瓶颈在 FD 还是 CPU」** 比报一个**假精确数字** 更稳。

---

### 18. 线上「消息慢 / 时延大」可能有哪些原因？

**答：**

- **网络路径**：**用户** 和 **机房间** 距离远、**跨洲**、**弱网**。

- **网关**：**Nginx 或 负载均衡** 若 **没按文档打开 WebSocket 的 proxy 设置**，**连接会** **异常、半残**、**时延**。

- **服务自身**：**Node 单进程** 若 **在事件循环里** 做 **CPU 重活**，**所有** 连接**都卡住**、**发得慢**。

- **你** 排障**可以** 说：我会**查** 网关日志、**看** 实例 CPU 与**连接数**、**看** 消息体是否**大** 到不该走 ws。

---

### 19. 为什么要控制推送频率、合并消息？

**答：**

- **服务器** 若 **每毫秒** 都 **给每个浏览器** 推一条**小消息**，**前端** 会 **频繁** **重排界面**、**网络** 也**撑不住**；**所以** 会 **做节流、合并、只推 ID 与摘要**，**大正文** 让客户端 **用 HTTP 再拉**。

---

## 六、安全

### 20. 生产环境为何常用 `wss://`？

**答：**

- **普通 HTTP 是明文的**；**HTTPS** 是 **HTTP + TLS**；**WSS** 是 **在 TLS 上承载 WebSocket 帧**。

- **谁** 用 **WSS**？**你的浏览器** 在 **https 的页面** 里 **一般必须** 用 **wss** 去连，否则**混合内容** 会被**浏览器** **拦截**；**攻击者** 在链路上**也就** 难以**窃听、篡改** 消息内容（**在 TLS 正常实现的前提下**）。

---

### 21. 有了鉴权，客户端是否可以在消息体里写「我是用户 5」就冒充用户 5？

**答：**

- **不能这样信。** **发送者** 的 **userId** 必须 **只** 能 **从** 服务端 **在 HTTP 里** 根据 **已登录会话** 取出来的 **`req.user.id`**（**或** 在**纯 socket 发消息** 的设计里，**从** 握手**已经绑在** `socket.data` 的 **id** 取）**来** 作为**入参** 写进数据库。

- **本仓库** 的 **`sendMessage`** **明确** 写 `senderId = req.user.id`，**不会** 用 **Body 里** 的「自称 id」**当真相**（若 Body 里还传了 senderId 也应**忽略**或**比对**后拒绝）。

---

### 22. CORS 对 WebSocket 有什么影响？

**答：**

- **浏览器的同源策略** 会检查 **从哪个网页页面** 发起的**握手**；**服务器** 会收到 **`Origin` 头**，**要** 决定 **是否** 允许**这个来源** 建连；**不** 允许**就** 握手失败、**连不上**。

- **这** 和 **HTTP 的 CORS 名单** 是**同一大类** 的**安全策略**；**你项目** 里 **Socket** 的 **cors** 与 **HTTP** 的 **cors** **都在** 配置里**要** 对齐**允许的前端域名**。

## 七、与消息队列、MQTT 的边界（中高级）

### 23. 有了 Kafka / Rabbit，还需要 WebSocket 吗？

**答：**

- **Kafka 解决的是** **服务与服务之间、异步、削峰、可靠投递** 的**消息**；**浏览器** **不会** 也去 **直连接 Kafka 协议**（**不现实也不安全**）。

- **所以** 常见架构是：**写库/写消息** 后，**发一条事件到队列**；**一个推送服务** 从队列里 **消费** 后，**再** 用 **本机** 的 **WebSocket 连接** 把**通知** **推到用户浏览器**。

- **结论**：**需要**；**分工不同**，**不是** 二选一。

---

### 24. MQTT 和 WebSocket 有什么区别？什么时候用 MQTT？

**答：**

- **MQTT** 是**一套** 面向**物联网** 的**发布/订阅** 协议，**有** **Broker**；**设备** 用 **小带宽** 连接 **Broker**；**网页** 有时用 **在 WebSocket 上承载 MQTT 包** 的方式连。

- **你做的网页聊天** 多数 **用 HTTP + Socket.IO/原生 WebSocket 就够**；**选 MQTT** 的**常见理由** 是**已有物联网体系、要统一** **大量设备+少量浏览器** 的**同一套** **Broker**。
